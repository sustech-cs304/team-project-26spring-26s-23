# 2026-04-11 网站文档翻修与重构设计

## 文档定位

本文档用于将已获确认的网站文档翻修设计正式落档。范围仅覆盖 Docusaurus 站点的文档信息架构、页面分工、内容迁移方式、写作规范、验收标准与风险约束，不包含实施结果，不展开任务拆解，也不修改除 [`docs/plans/2026-04-11-website-docs-refresh-design.md`](docs/plans/2026-04-11-website-docs-refresh-design.md) 之外的任何文件。

## 背景与问题

当前文档站点已经具备基础内容，但主阅读路径仍然偏向按技术分册组织，读者更容易从旧的 system / frontend / backend 结构进入，而不是从自己的任务出发进入。结合现状，可以看到以下问题：

- 当前站点配置位于 [`website/docusaurus.config.ts`](../website/docusaurus.config.ts:7)，文档源目录位于 [`docs/`](../README.md:1)；站点本身具备重组条件，但现有导航没有把读者按目标任务清晰分流。
- 当前侧边栏以 [`website/sidebars.ts`](../website/sidebars.ts:15) 的自动生成策略为主，导航更接近按目录暴露内容，而不是按读者问题组织内容。
- 旧文档中大量内容仍按 [`docs/system/`](../system/architecture-overview.md:1)、[`docs/frontend/`](../frontend/README.md:1)、[`docs/backend/`](../backend/README.md:1) 这样的技术分册展开，普通使用者很难快速找到上手路径。
- 一些高变化主题缺少唯一事实来源，容易在不同页面重复描述，进一步放大过时风险。
- 历史文档中存在已过时口径，例如把 `session/create` / `message/send` 写成主链、把 active provider 写成当前配置中心、把 thinking 仍写成旧 intent / 档位、把 CopilotKit 写成当前运行时依赖。这些内容如果继续出现在主路径，会误导读者对当前实现的理解。

问题的本质不是文档数量不足，而是信息架构没有围绕当前两类核心读者重新设计：

1. 普通使用者需要尽快知道产品是什么、怎么开始、怎么配置、能做什么、当前限制是什么。
2. 开发者需要尽快知道本地怎么跑、系统如何组织、运行时怎么工作、配置和状态如何流动、代码落点在哪。

如果继续沿用旧的技术分册作为主入口，普通使用者会被内部术语拖慢，开发者也会在过时页面和真实代码之间来回对照，阅读成本高且容易误判。

## 目标与非目标

### 目标

1. 在同一个站点内建立双入口，让首页与主导航先分流到“给使用者”和“给开发者”。
2. 建立共享事实层，集中承载高变化、跨页面复用的唯一事实来源。
3. 允许对 [`docs/`](../README.md:1) 的页面结构与 URL 做大幅重组，不再以旧的 system / frontend / backend 技术分册作为主阅读入口。
4. 让使用者路径可以在 3 到 5 页内完成“这是什么 → 怎么开始 → 怎么配置 → 能做什么 / 有什么限制”的核心认知闭环。
5. 让开发者路径可以在 4 到 6 页内完成“怎么跑 → 架构概览 → 聊天运行时 → 配置状态 → 前后端落点”的核心认知闭环。
6. 统一写作口径为说明书式中文，保证大多数开发者和普通使用者都能读懂。
7. 明确旧文档只作为素材来源，当前事实以代码、测试、近期设计与实现记录为准。

### 非目标

1. 本文档不直接实施站点改版。
2. 本文档不逐页保留旧信息架构，也不承诺与旧 URL 完全兼容。
3. 本文档不编写最终页面正文，只定义应如何组织与重写。
4. 本轮不把历史分析、会议记录、旧草案类内容纳入主阅读路径。
5. 本轮不扩展新的产品能力范围，只对齐当前项目实际进度与现状表述。

## 事实基线

本轮设计必须建立在当前更可靠的事实基线上，而不是沿用历史文档的旧说法。已确认的事实基线如下：

### 可靠事实

- 当前项目以桌面宿主为主要运行形态。
- 配置模型采用双层配置思路，而不是单一中心化配置页心智。
- 聊天能力以 thread / run 流式链路为当前可靠主线。
- 项目具备多 provider catalog 能力。
- thinking 系列能力已经存在，并且应作为当前能力说明的一部分。
- 设置工作区扩展属于当前可靠事实的一部分。
- Blackboard / TIS 领域能力已经存在，但尚未完全产品化，表达时必须保留边界感。

### 近期容易过时的旧口径

以下说法不得继续作为主路径事实：

- 把 `session/create` / `message/send` 说成当前主链。
- 把 active provider 说成当前配置中心。
- 把 thinking 仍写成旧 intent / 档位体系。
- 把 CopilotKit 写成当前运行时依赖。

### 事实优先级

当旧文档与当前实现不一致时，判定顺序必须为：

1. 代码
2. 测试
3. 近期设计 / 实现记录
4. 旧文档

这意味着 [`docs/system/`](../system/architecture-overview.md:1)、[`docs/frontend/`](../frontend/README.md:1)、[`docs/backend/`](../backend/README.md:1) 仅作为素材来源，不再自动被视为权威来源。

## 为什么采用双入口 + 共享事实层

### 核心原因

采用“双入口 + 共享事实层”，是为了同时解决两类问题：

1. 入口问题：不同读者进入站点时，关心的问题完全不同。
2. 事实问题：高变化主题若分散在多页重复书写，会迅速过时并相互冲突。

### 对普通使用者的价值

普通使用者并不需要先理解 system、frontend、backend 的技术边界。他们首先需要的是任务路径，而不是实现分层。双入口能让这类读者从首页就进入“给使用者”的路径，优先看到产品定义、上手动作、配置方法、能力边界与限制说明，避免被内部实现术语打断。

### 对开发者的价值

开发者需要更强的结构感与代码对照能力，但他们同样不适合从历史技术分册碎片化地开始阅读。双入口能把开发者直接导向一条按上手顺序组织的路径：先运行，再看架构，再理解运行时，再理解配置与状态，再定位前后端代码落点。

### 对事实一致性的价值

Provider、thinking、术语表、能力边界、运行时接口 / 事件这类主题变化快、引用多。如果这些内容分别散落在使用者页和开发者页中，会出现多个版本同时存在的情况。共享事实层的作用，就是把这些高变化主题收口为唯一事实源，其他页面只链接引用，不再多处复制。

### 结论

双入口负责把人带到对的阅读路径，共享事实层负责保证关键事实只有一份权威来源。这一组合既能服务普通使用者，也能服务开发者，同时降低过时口径扩散的风险。

## 目标信息架构

## 总体结构

目标站点结构建议重组为三层：

1. 面向使用者的任务路径。
2. 面向开发者的工程路径。
3. 面向两类读者共享的事实层。

建议目录如下：

- [`docs/users/`](../users)
- [`docs/developers/`](../developers)
- [`docs/reference/`](../reference)

### 信息架构原则

- 首页与主导航先做读者分流，再进入各自路径。
- 不再让读者直接从旧的技术分册切入。
- 高变化主题统一进入共享事实层。
- 各主路径页面只保留本页面应承担的解释职责，不重复展开共享事实。
- 每页都必须显式标注当前状态，且只允许使用“已可用 / 部分接通 / 规划中”三档口径。

1. 产品是什么。
2. 5 分钟上手。
3. 模型与 Provider 配置。
4. 聊天与 thinking 能力怎么用。
5. 当前能力边界与已知限制。

建议页面地图如下：

| --- | --- | --- |
| 产品是什么 | [`docs/users/overview.md`](../users/overview.md) | 用非内部术语说明产品定位、适用场景、当前状态范围 |
| 5 分钟上手 | [`docs/users/quickstart.md`](../users/quickstart.md) | 用最短路径说明首次启动、基础操作与预期结果 |
| 模型与 Provider 配置 | [`docs/users/configure-models.md`](../users/configure-models.md) | 只解释用户真正需要做的配置动作，复杂事实链接到共享页 |

共享事实层不承担主阅读路径导流职责，它承担的是唯一事实源职责。使用者页和开发者页只在需要时链接引用，而不复制其全文。

## 导航与 URL 设计原则

### 首页与导航

首页和顶层导航必须一眼分流出“给使用者”和“给开发者”。这不是视觉装饰要求，而是信息架构是否成立的核心判据。

要求如下：

- 首页首屏必须先给出两类入口。
- 顶层导航必须直接出现“给使用者”和“给开发者”。
- 共享事实层可以作为独立入口出现，但不能抢占主入口位置。
- 旧的 system / frontend / backend 分册不再作为首层入口。

### URL 原则

用户已确认允许大幅重组 [`docs/`](../README.md:1) 的页面结构与 URL，因此 URL 设计应服务新的阅读路径，而不是迁就旧目录。

建议原则如下：

- URL 应直接反映读者路径，例如 `/users/...`、`/developers/...`、`/reference/...`。
- 旧 URL 是否保留兼容，不作为本轮设计前提。
- 页面命名优先表达用途，不优先表达技术模块归属。

## 内容迁移策略

### 总体迁移方式

本轮不是在旧页面上逐页打补丁，而是把 [`docs/system/`](../system/architecture-overview.md:1)、[`docs/frontend/`](../frontend/README.md:1)、[`docs/backend/`](../backend/README.md:1) 的现有事实重新分类后重写。

迁移时必须遵守以下原则：

1. 先按新信息架构分类，再决定内容去向。
2. 旧页面只作为素材来源，不作为结构来源。
3. 与当前实现不一致的旧内容必须舍弃或重写，不能为保留历史痕迹而继续进入主路径。
4. 对于 Provider、thinking、术语、能力边界、运行时事件等高变化内容，优先抽出进入共享事实层。
5. 历史分析、会议记录、旧草案类内容降权，不进入主阅读路径。

### 迁移分流规则

| 原内容类型 | 目标去向 | 处理原则 |
| --- | --- | --- |
| 面向用户的动作说明 | [`docs/users/`](../users) | 改写为用户动作与结果导向 |
| 面向开发者的架构与代码说明 | [`docs/developers/`](../developers) | 按上手顺序重排，而不是按旧模块分册搬运 |
| 高变化共用事实 | [`docs/reference/`](../reference) | 收口为唯一事实源，其他页面仅链接 |
| 历史记录、分析、草案 | 非主路径区域 | 降权保留或单独归档，不进入主阅读流 |

### 迁移优先级

本轮优先完成以下工作方向：

1. 信息架构重建。
2. 核心入口页重写。
3. 关键共享事实页收口。

这意味着即使旧文档还有很多历史材料未整理，只要不影响新的主阅读路径，就不应阻塞本轮站点翻修。

## 写作规范

### 页面开头固定交代项

每页开头必须先回答三个问题，再进入正文：

1. 这页给谁看。
2. 这页能解决什么问题。
3. 这页当前覆盖到哪。

这样做的目的，是让读者在进入细节前先确认自己是否来对页面，也让页面边界保持稳定。

### 使用者文档写法

使用者文档应坚持以下规则：

- 只写用户真正关心的动作和结果。
- 尽量不把内部实现术语放进主叙事。
- 优先解释做什么、看到什么、下一步是什么。
- 需要引用复杂事实时，链接到共享事实页，而不是在用户页重复长解释。

### 开发者文档写法

开发者文档应坚持以下规则：

- 按上手顺序组织，而不是按组织架构或历史目录组织。
- 先运行，再架构，再机制，再代码落点。
- 说明实现时，优先给出结构关系与定位入口，再给细节。
- 引用高变化主题时，链接到共享事实页，避免复制粘贴。

### 共享事实页写法

共享事实页应坚持以下规则：

- 每个主题只保留一份权威表达。
- 重点写定义、边界、状态与引用关系。
- 不承担完整教程职责。
- 被主路径页引用时，主路径页只写“何时需要看它”，不复制事实正文。

### 状态表达规范

所有页面都必须显式使用以下三档状态口径：

- 已可用
- 部分接通
- 规划中

不得混用其他含糊表述作为主状态，例如“基本支持”“即将可用”“实验性但差不多可用”等。

### 语言风格规范

统一采用说明书式中文，要求如下：

- 短句。
- 先结论后解释。
- 少缩写。
- 少黑话。
- 尽量给具体例子。
- 不把实现细节堆成抽象名词串。

目标不是写成营销文案，也不是写成论文，而是让绝大多数开发者与普通使用者都能快速看懂。

## 验收标准

### 信息架构验收

- 首页与侧边栏必须能一眼分流出“给使用者”和“给开发者”。
- 读者不应再被迫从旧的 system / frontend / backend 技术分册进入主路径。
- 共享事实层必须作为独立且明确的事实层存在。

### 使用者路径验收

使用者路径应能在 3 到 5 页内完成以下认知闭环：

1. 这是什么。
2. 怎么开始。
3. 怎么配置。
4. 能做什么。
5. 有什么限制。

如果读者需要先读架构页、前端页或后端页才能完成基本理解，则说明路径设计失败。

### 开发者路径验收

开发者路径应能在 4 到 6 页内完成以下认知闭环：

1. 怎么跑。
2. 架构概览。
3. 聊天运行时。
4. 配置状态。
5. 前后端落点。

如果读者必须先穿过旧目录树才能建立整体理解，则说明路径设计失败。

### 共享事实层验收

以下高变化主题必须由共享页承担唯一事实源职责：

- Provider
- thinking
- 术语表
- 能力边界
- 运行时事件

其他页面可以引用这些主题，但不应各自复制一份版本。

### 状态表达验收

- 所有页面都要显式使用“已可用 / 部分接通 / 规划中”表达当前状态。
- 不允许同一能力在不同页面出现冲突状态表达。

### 范围验收

- 本轮优先完成信息架构重建、核心入口页重写、关键共享事实页收口。
- 历史分析、会议记录、旧草案类内容降权，不进入主阅读路径。

## 风险与注意事项

### 1. 双入口建立后仍残留技术分册主心智

如果首页虽然新增了“给使用者”和“给开发者”，但主导航、侧边栏或核心页面仍让旧的 system / frontend / backend 分册占据主入口，用户仍会回到旧心智。控制原则是：双入口必须先于技术分册出现，且主路径不能依赖旧分册成立。

### 2. 共享事实层没有真正成为唯一事实源

如果 Provider、thinking、能力边界、运行时事件等内容继续在多页重复出现，过时风险不会下降，只会从旧目录复制到新目录。控制原则是：共享事实页只保留一份权威表述，其他页面链接引用。

### 3. 使用者路径被内部术语污染

如果使用者页继续大量出现 provider route、runtime event、状态模型等内部表达，双入口会名义成立、实际失效。控制原则是：使用者文档只写动作与结果，把内部术语下沉到共享事实页或开发者页。

### 4. 开发者路径仍按历史模块拼接

如果开发者路径只是把旧的 frontend / backend / system 页面重新挂到一个新目录里，而没有改成“先运行、再架构、再机制、再代码落点”的阅读顺序，开发者仍然需要自行重建心智模型。控制原则是：开发者路径必须按上手顺序重写，而不是按旧章节重命名。

### 5. 状态口径继续混乱

如果页面继续混用“支持中”“实验性”“已接入一部分”“未来考虑”等说法，读者很难判断实际进度。控制原则是：统一使用“已可用 / 部分接通 / 规划中”。

### 6. 历史材料重新挤入主路径

如果为了保留历史积累，把旧分析、会议记录、旧草案重新塞进主路径，会拖慢阅读节奏，也会提高过时内容暴露概率。控制原则是：历史材料降权处理，不进入主阅读流。

## 结论

本轮网站文档翻修的核心结论是：站点应从旧的技术分册主入口，重构为“给使用者”“给开发者”“共享事实层”三层信息架构。其中，双入口负责把不同读者快速导向正确路径，共享事实层负责承载 Provider、thinking、术语表、能力边界、运行时接口 / 事件等高变化主题的唯一事实源。

在此结构下，[`docs/users/`](../users) 承担普通使用者路径，[`docs/developers/`](../developers) 承担开发者路径，[`docs/reference/`](../reference) 承担共享事实层；旧的 [`docs/system/`](../system/architecture-overview.md:1)、[`docs/frontend/`](../frontend/README.md:1)、[`docs/backend/`](../backend/README.md:1) 不再作为主阅读入口，只作为重写时的素材来源。后续实施必须围绕这一结构展开，优先完成信息架构重建、核心入口页重写与关键共享事实页收口，并以统一状态口径和说明书式中文完成内容重写。
