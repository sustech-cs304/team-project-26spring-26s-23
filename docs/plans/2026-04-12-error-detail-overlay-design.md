# 2026-04-12 聊天失败卡片错误详情叠加层设计

## 文档定位

本文档用于将已确认的聊天失败卡片错误详情叠加层方案正式落档。范围仅覆盖聊天区失败卡片的错误详情展示设计、信息架构、数据适配约束、复制策略、视觉与交互原则、测试与验收口径，不包含代码实现，不展开实现计划，也不修改 Markdown 以外的文件。

## 背景与问题

当前聊天区中的失败卡片需要同时满足两类目标：

1. 默认态必须保持简洁，避免把诊断细节直接堆在消息流中。
2. 当用户或开发者需要排查问题时，界面又必须能够提供足够完整、可复制、可转交的错误详情。

现状中的主要问题是：

- 失败卡片默认文案必须克制，但这会使部分失败在消息流中只剩一句总结，无法承接进一步排查。
- 不同失败来源拥有的信息丰富度并不一致，尤其是流式失败通常已有较完整结构化字段，而发送前校验失败与 `run/start` 请求失败若不额外保留字段，进入聊天消息后可能只剩简短文案。
- 若直接为每种失败类型硬编码固定详情布局，会让后续错误来源扩展变得脆弱，也会破坏不同失败场景之间的阅读一致性。
- 当前聊天链路中已有失败文案映射与消息列表渲染路径，但尚未形成一套统一的错误详情视图模型，导致详情展示难以稳定复用。

因此，本轮需要在保持默认失败卡片简洁的前提下，引入一个统一、极简、可动态分块的错误详情叠加层，使三类失败来源都能通过同一套展示语义被查看、复制与转交。

## 目标与非目标

### 目标

1. 首版只处理聊天区里的失败卡片，并在现有失败卡片上增加一个 `详情` 图标按钮作为入口。
2. 首版覆盖聊天区所有失败卡片，包括发送前校验失败、`run/start` 请求失败、流式运行失败。
3. 用户点击详情按钮后，打开一个居中的极简叠加层 / dialog，用统一视觉和统一阅读节奏承载当前失败的完整诊断信息。
4. 叠加层内部按照当前失败实际拥有的信息组动态生成多个信息面板块，而不是固定块数，也不是按失败类型硬编码单独布局。
5. 定义稳定的分块排序原则，使不同失败类型即使块数不同，也能保持一致的阅读顺序。
6. 为叠加层定义总复制与单块复制能力，复制结果优先输出结构化纯文本摘要，便于用户贴给开发者或 issue。
7. 在错误渲染链上引入统一的错误详情视图模型，将三类失败来源整理成同一套可分组数据输入，再由聊天消息列表中的详情层组件按组渲染。
8. 明确流式失败优先复用现有 terminal message 字段，同时要求发送前校验失败与请求启动失败在进入聊天消息时额外保留可用结构化字段。
9. 用极简主义原则约束视觉与交互：默认态克制、展开后信息充分，质感来自排版秩序、留白与层级，而不是厚重装饰。

### 非目标

1. 本文档不包含代码实现。
2. 本文档不展开实现计划。
3. 本轮不处理聊天区之外的错误展示入口。
4. 本轮不改造失败卡片的默认简洁文案策略，默认态仍保持克制。
5. 本轮不为每种失败类型设计完全不同的定制化详情界面。
6. 本轮不要求首版支持复杂筛选、搜索、折叠树、语法高亮或多层诊断导航。
7. 本轮不要求补齐不存在的底层诊断数据；当某类错误没有更多结构化字段时，详情层仍可打开，但只展示克制的空状态说明。

## 交互入口与信息架构

### 默认失败卡片

默认失败卡片继续保持简洁文案，不在消息流中直接展开长错误信息。首版只在现有失败卡片上新增一个 `详情` 图标按钮，作为进入错误详情叠加层的唯一主入口。

该入口的设计约束如下：

- 必须依附于现有失败卡片，不新增额外列表行或独立调试区。
- 按钮语义明确，但视觉权重低于失败摘要本身。
- 默认卡片仍以简洁失败结论为主，不因详情能力加入而膨胀为重诊断卡片。

### 详情层打开后的整体结构

点击详情按钮后，打开一个居中的极简叠加层 / dialog。交互语义可参考现有设置页 dialog，但视觉上更克制、更安静。叠加层的信息架构分为三层：

1. 顶部区：当前失败的简洁标题、必要的一句话说明、关闭动作、总复制按钮。
2. 主体区：根据当前失败实际拥有的信息组动态生成多个信息面板块。
3. 空状态区：当某些失败没有更多结构化字段时，仍提供统一详情层与克制说明，而不是拒绝打开。

### 组件复用边界

本轮已确认以下复用方向，需要在实现阶段延续：

- dialog 语义与基础交互参考现有设置页 dialog。
- 图标按钮样式参考全局 controls 的 icon-button 模式。
- 复制交互参考 provider secret copy 的现有交互模式。

本设计只定义复用方向与设计语义，不要求在文档中展开组件级实现细节。

## 动态分块规则

### 核心原则

叠加层内部不固定为 5 块，也不按失败类型硬编码固定版式，而是根据当前失败类型实际拥有的信息组动态生成多个信息面板块。这些块代表同一次失败中的不同信息维度切分，而不是某类失败专属的静态模板。

换言之，决定块数的是 `当前错误拥有了哪些可展示信息组`，而不是 `这是哪一种错误类型`。

### 信息组的稳定排序原则

为保证不同失败类型虽然块数不同，但阅读节奏一致，所有动态生成的信息块必须遵循以下稳定排序原则：

1. 先放用户最易理解的摘要类信息。
2. 再放请求或运行上下文信息。
3. 再放工具与模型上下文信息。
4. 最后才放原始详情。

该排序原则的目标不是穷举所有字段，而是固定用户阅读路径：先看发生了什么，再看发生在什么请求或运行阶段，再看涉及哪些工具或模型上下文，最后才接触原始错误细节。

### 建议信息组类别

首版建议使用以下抽象信息组类别来组织详情层，具体是否出现取决于当前错误是否提供对应数据：

| 信息组类别 | 排序层级 | 作用 | 典型内容示例 |
| --- | --- | --- | --- |
| 摘要 | 最高优先级 | 用用户最容易理解的方式概括本次失败 | 错误标题、简短解释、失败阶段、错误码摘要 |
| 请求 / 运行上下文 | 中优先级 | 说明失败发生在哪个动作、哪个阶段、哪次请求 | `stage`、`requestedMethod`、运行阶段、请求动作 |
| 工具 / 模型上下文 | 次低优先级 | 展示与本次失败关联的模型、工具、路由或能力上下文 | 模型标识、工具名、路由信息、能力上下文 |
| 原始详情 | 最低优先级 | 保留原始错误正文、原始细节或未归类诊断内容 | `details`、原始 message、序列化诊断文本 |

这里的类别是展示语义，不等同于底层固定字段对象。实现阶段应允许多个底层字段被归并到同一组，也允许某些错误只生成其中一到两组。

### 分块生成约束

每个信息块都必须包含以下要素：

- 标题
- 简短说明
- 内容区
- 单独复制按钮

其中：

- 标题负责说明当前块的阅读主题。
- 简短说明负责告诉用户这块信息能帮助判断什么，不承担长篇解释职责。
- 内容区负责承载结构化字段、列表、键值对或克制纯文本。
- 单独复制按钮负责把当前块复制成独立纯文本摘要。

### 缺少数据时的处理

若某类错误没有更多结构化字段，也仍然允许打开详情层。此时可出现以下两种情况：

1. 只有摘要块，没有更多上下文块。
2. 连摘要之外的结构化内容都缺失，需要在相应位置显示 `暂无更多详情` 或等价的克制空状态说明。

空状态必须清晰但克制，不应给用户造成界面损坏或缺陷感，也不应伪造不存在的结构化信息。

## 复制策略

### 总复制

叠加层顶部提供一个总复制按钮，用于复制当前错误的完整诊断摘要。总复制内容优先输出结构化纯文本摘要，而不是面向界面排版的碎片文本。

总复制应遵循以下原则：

- 复制结果适合直接贴给开发者、测试同学或 issue。
- 输出尽量保留清晰层级与字段标签，避免复制后变成不可读的长串文本。
- 当某些信息组缺失时，不应输出空洞占位内容；只汇总当前错误真实拥有的信息。
- 若存在原始详情，也应在总复制中保留其位置，但放在摘要与上下文信息之后。

### 单块复制

每个信息块都有自己的复制按钮，复制该块的结构化纯文本摘要。单块复制的意义在于：

- 用户可以只复制最有价值的局部信息。
- 开发排障时可以单独转发某一维度内容，而不必附带全部错误文本。
- 对于原始详情块，可以单独复制原始内容，避免被额外摘要化改写。

### 纯文本摘要格式原则

复制输出优先使用结构化纯文本摘要，建议遵循以下原则：

- 以块标题为一级分隔。
- 块内使用稳定、清晰的键值或短列表表达。
- 用词与界面标题尽量一致，减少复制前后语义跳变。
- 避免混入界面装饰字符、视觉标签或仅服务于 UI 的冗余文案。
- 对原始详情块保留原始内容特性，不强行压缩成过度摘要。

## 视觉与交互原则

### 极简主义总原则

本设计必须符合 Minimalism 家族设计原则。具体要求如下：

- 默认态克制，信息不外溢到消息流。
- 展开后信息充分，但仍保持安静、稳定、低打扰。
- 界面质感来自排版秩序、留白与层级，而不是厚重边框、强阴影、复杂背景或炫目动画。

### 叠加层视觉方向

叠加层视觉上应体现以下特征：

- 居中呈现，聚焦当前错误本身。
- 使用中性色体系，而不是高饱和强调色堆叠。
- 使用细线、极弱阴影和留白建立层次。
- 信息面板块之间依靠节奏化间距和排版分组来区分，而不是依赖厚重容器感。
- 反馈轻量，避免强烈弹跳、夸张过渡或过度动画。

### 交互语义

- dialog 的打开、关闭、遮罩与焦点管理语义参考现有设置页 dialog。
- `详情` 图标按钮、顶部总复制按钮、单块复制按钮都应提供一致、克制的反馈。
- 复制反馈要清晰，但不应打断阅读节奏。
- 当错误信息较多时，界面应优先保障阅读顺序和内容可扫描性，而不是制造复杂交互层级。

## 数据适配方案

### 统一错误详情视图模型

首版需要在错误渲染链上引入统一的错误详情视图模型。该视图模型的职责不是替代现有所有错误对象，而是在错误进入聊天消息渲染时，把不同来源的错误整理成同一套可分组数据输入，供详情层组件直接按组渲染。

这意味着：

- 发送前校验失败、`run/start` 请求失败、流式运行失败最终都需要被适配成统一的详情层输入结构。
- 详情层组件不应关心底层错误来自哪条链路，而只关心当前有哪些信息组、各组标题与内容是什么、复制文本是什么。
- 失败类型差异保留在数据适配阶段，而不泄漏到展示层的组件分支膨胀中。

### 三类失败来源的数据要求

#### 1. 流式运行失败

流式失败优先复用当前 terminal message 已有字段。已确认这一路径通常已具备相对完整的结构化信息，因此应优先从现有字段中整理出摘要、运行上下文、工具 / 模型上下文与原始详情等可展示信息组。

#### 2. 发送前校验失败

发送前校验失败若仅保留一句失败文案，详情层会失去可用价值。因此，这类错误在进入聊天消息时，需要额外保留可用的结构化字段，至少包括文档中已确认的 `code`、`details`、`stage`、`requestedMethod` 等可用字段或其等价信息，以便后续适配为详情组。

#### 3. `run/start` 请求失败

`run/start` 请求失败同样需要在错误进入聊天消息时额外保留可用结构化字段，避免详情层只能重复默认失败卡片上的一句简短文案。其适配目标与发送前校验失败一致：保证在统一视图模型中仍能生成足够的摘要和上下文信息组。

### 数据流边界

本轮设计明确以下数据流原则：

1. 失败卡片默认文案与详情层输入分离。默认文案继续服务于消息流简洁性，详情视图模型服务于展开后的信息充分性。
2. 统一视图模型应在错误渲染链上尽早形成，避免详情层组件在渲染期直接拼接底层多源错误对象。
3. 对于没有更多数据的失败，也要生成一个可以打开详情层的最小输入，使空状态成为设计的一部分，而不是异常分支。

## 信息块内容建议

为确保不同失败来源在统一视图模型中具备可读性，首版可按照以下思路组织每类块的内容：

### 摘要块

- 本次失败的简洁标题
- 一句话说明
- 可理解的失败阶段摘要
- 若存在，可展示短错误码或短原因标签

### 请求 / 运行上下文块

- 当前动作或请求类型
- `requestedMethod`
- 运行阶段或失败进入点
- 与本次会话或执行链有关的必要上下文

### 工具 / 模型上下文块

- 模型标识或模型相关上下文
- 涉及的工具、能力或路由
- 对理解失败来源有帮助，但不应喧宾夺主的执行环境信息

### 原始详情块

- 原始错误详情文本
- 原始 `details`
- 未经摘要化的底层诊断内容

原始详情块永远排在最后，避免用户一进入就面对未经整理的大段原文。

## 测试与验收

### 验收标准

本轮设计的验收标准已确认如下：

- 默认失败卡片仍然简洁。
- 三类失败来源都能打开统一详情层。
- 详情层会根据当前错误拥有的信息动态生成块。
- 总复制与单块复制可用。
- 缺少数据时给出清晰但克制的空状态说明。

### 验证场景

至少应按以下场景验证：

1. 发送前校验失败：确认失败卡片默认态简洁，打开后可看到统一详情层；若有结构化字段，则生成对应信息块；若字段有限，则出现克制空状态。
2. `run/start` 请求失败：确认失败卡片与详情层入口一致；详情层能展示摘要与请求上下文；复制结果可直接用于反馈问题。
3. 流式运行失败：确认可复用现有 terminal message 字段形成更完整分块，并保持统一排序与复制体验。
4. 缺少原始详情的失败：确认仍能打开详情层，不出现空白容器或错误布局。
5. 存在原始详情的失败：确认原始详情块排在最后，且可被单独复制原始内容。

### 复核重点

实现完成后，验收时需要重点复核以下维度：

- 默认消息流是否仍保持简洁，没有把诊断层内容前置到卡片主视图。
- 动态分块是否由信息组驱动，而非由失败类型驱动固定块数。
- 排序原则是否稳定执行，未因某类失败而打乱阅读节奏。
- 总复制与单块复制是否都输出结构化纯文本摘要。
- 发送前校验失败与请求失败是否已真正保留额外结构化字段，而不是仍只剩一句文案。

## 风险与后续扩展

### 风险

#### 1. 数据不足导致详情层形式化存在

若发送前校验失败与 `run/start` 请求失败在进入聊天消息时仍未保留足够字段，详情层将退化为只重复失败卡片文案。控制原则是：必须在错误进入聊天消息时额外保留 `code`、`details`、`stage`、`requestedMethod` 等可用字段或等价信息。

#### 2. 按失败类型硬编码块布局

若后续实现直接按三类失败分别写死块结构，会导致扩展成本上升，也会破坏统一阅读节奏。控制原则是：使用统一视图模型与信息组驱动分块，失败类型只影响输入数据，不直接决定展示骨架。

#### 3. 视觉实现偏离极简主义

若叠加层依赖厚重边框、强阴影、复杂背景或强调色堆叠，会偏离已确认的 Minimalism 家族方向。控制原则是：通过留白、细线、排版层级和轻反馈建立质感。

#### 4. 复制输出沦为原始字段倾倒

若复制功能直接输出未经整理的底层对象，用户虽然获得了信息，但难以高效转发。控制原则是：总复制与单块复制都优先输出结构化纯文本摘要；仅原始详情块允许保留更接近原文的内容。

#### 5. 空状态处理不克制

若无数据时直接禁用详情入口，会让不同失败来源的交互断裂；若空状态文案过强，又会破坏极简体验。控制原则是：保留统一入口与详情层容器，用清晰但克制的说明承接低数据量失败。

### 后续扩展方向

以下方向可作为后续扩展，但不属于本轮范围：

- 将统一错误详情视图模型推广到聊天区之外的错误入口。
- 在保留极简基调的前提下，为更复杂的执行失败引入更丰富的信息组。
- 在验证首版稳定后，进一步沉淀统一错误复制模板，服务于 issue、反馈单与开发排障。

## 结论

本轮设计已经明确：聊天区失败卡片继续保持简洁默认态，并在现有卡片上新增一个低权重的 `详情` 图标按钮作为统一入口；首版覆盖发送前校验失败、`run/start` 请求失败与流式运行失败三类来源；点击后打开居中的极简叠加层 / dialog，并按当前错误真实拥有的信息组动态生成多个信息块。

所有信息块遵循统一排序原则，从摘要到请求 / 运行上下文，再到工具 / 模型上下文，最后到原始详情；每块都具备标题、简短说明、内容区与单独复制按钮，叠加层顶部另有总复制按钮。复制结果优先输出结构化纯文本摘要，以便用户转交开发者或 issue。

为支撑这一展示能力，错误渲染链需要引入统一的错误详情视图模型。流式失败优先复用现有 terminal message 字段；发送前校验失败与请求启动失败则必须在进入聊天消息时额外保留结构化字段，避免详情层只剩一句文案。后续实现与验收必须以本文档为唯一设计依据，并严格遵守极简主义视觉与交互原则。
