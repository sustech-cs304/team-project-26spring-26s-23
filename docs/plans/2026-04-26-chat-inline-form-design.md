# 聊天内联表单设计

## 背景与目标

当前需求面向聊天主线，而不是设置页。目标是在 AI 对话过程中，由 AI 通过受控工具触发一个内联表单，用来向用户收集当前回合继续推进所需的信息。表单需要直接嵌入聊天消息流，用户完成填写后继续当前线程，不跳出既有对话上下文。

这项设计与现有聊天主链保持一致，语义上应依附当前的 `thread/run` 与事件流模型，而不是额外建立一套脱离消息流的页面机制。相关背景可以先对照 [聊天运行时](../developers/chat-runtime.md)、[运行时接口 / 事件参考](../reference/runtime-events.md) 与 [聊天运行时 HTTP 契约（旧资料）](../system/chat-runtime-contract.md)。前端代码上下文可参考 [前端实现](../developers/frontend.md) 中提到的聊天入口，以及 `frontend-copilot/src/features/copilot/` 目录下现有消息流、消息列表与 run 片段处理相关模块。

本设计的直接目标有四点：

1. AI 只有在明确调用受控工具时，前端才渲染表单。
2. 表单以聊天消息中的内联段形式出现，并留在当前对话线程内。
3. 用户提交后，前端生成一条新的用户消息，将可见摘要与结构化 JSON 载荷一起送入后续对话。
4. `json-render` 只承担受控 schema 到 React 表单 UI 的渲染职责，不承担聊天协议、历史归档与运行时状态编排。

## 非目标

这份设计不覆盖下列内容：

1. 它不设计设置页表单，也不把聊天内联表单抽象成通用设置系统。
2. 它不允许 AI 直接输出任意表单 JSON 并由前端自由解释。
3. 它不重新定义当前聊天主协议，不替代 `thread/run` 主链。
4. 它不让 `json-render` 接触聊天原始消息结构、消息历史或运行时状态机。
5. 它不覆盖流式 patch、复杂嵌套布局、文件上传、动态增删字段、跨表单依赖、草稿恢复、secret 或高风险信息采集。

## 已选方案

当前已确认采用方案 A，也就是工具触发型内联表单。

核心约束非常明确：AI 不能直接在 assistant 文本中塞入一段任意表单描述并期待前端自行猜测；AI 需要调用受控工具，概念上类似 `request_user_form`，再由运行时生成结构化事件，前端根据该事件在聊天消息流里投影出表单段。

这意味着表单渲染入口应落在既有工具生命周期或工具结果链路中，而不是依赖 markdown 解析、文本约定或非结构化提示词。

## 用户流程

### 正常流程

1. 用户在当前线程中发起一个普通聊天请求。
2. AI 在推理过程中判断继续执行需要补充信息，于是调用受控表单工具。
3. 运行时生成结构化事件，并沿既有工具事件链路回传给前端。
4. 前端在当前 assistant 消息相关的消息流位置，投影出一个内联表单段。
5. 用户在表单中填写字段，并在本地完成基础校验。
6. 用户提交后，前端生成一条新的用户消息。
7. 这条用户消息同时包含可见摘要与结构化 JSON 载荷。
8. 当前线程继续沿既有聊天主链运行，后续 assistant 响应把这条表单提交视为新的用户输入。

### 提交后的展示

提交成功后，表单应锁定为只读状态，并继续保留在历史消息中用于回看。这样可以让同一线程中的上下文保持完整，也能避免用户误以为同一表单还能重复修改并再次提交。

### 失效流程

如果线程发生变化，或者当前表单对应的回合已经失效，前端应将表单标记为过期。过期表单不应继续允许提交，以免把不再属于当前对话时序的数据送入消息流。

## 总体架构

整体架构沿当前聊天主链展开，可以概括为四段：

1. AI 通过受控工具声明“需要一个表单”。
2. 运行时在工具生命周期或工具结果链路中发出结构化事件。
3. 前端把该结构化事件投影为聊天消息中的内联表单片段。
4. 用户提交后，前端再把表单结果写回为新的用户消息，并继续当前线程。

这套架构与 [运行时接口 / 事件参考](../reference/runtime-events.md) 中已有的 `tool_event` 语义最容易对齐，也与 [聊天运行时](../developers/chat-runtime.md) 对“工具目录、工具启用策略、工具生命周期进入同一条 run 事件流”的描述保持一致。

从职责划分看，关键边界如下：

1. AI 负责决定是否请求表单。
2. 运行时负责把受控工具调用转换为结构化事件。
3. 聊天前端负责把结构化事件纳入消息投影，并管理提交流程。
4. `json-render` 负责把规整后的 schema 渲染成具体表单 UI。

## 事件与数据流

### 触发阶段

触发的唯一入口是 AI 明确调用某个受控工具或动作。当前不接受以下做法：

1. 从 assistant markdown 文本中猜测是否有表单。
2. 让 AI 自由拼装任意 JSON 并交给前端盲目渲染。
3. 在聊天协议之外另开一条表单专用前端状态通道。

运行时应在既有工具生命周期或工具结果链路中生成一类结构化事件。文档层面不强行规定精确字段名，但需要明确它至少表达这些信息：

1. 这是一条受控表单请求事件。
2. 它隶属于当前线程和当前 run 的既有上下文。
3. 它携带规整后的表单 schema 或可供表单协议适配层消费的结构化数据。
4. 它具备足以判断过期、只读与提交流程归属的标识信息。

### 渲染阶段

前端接收结构化事件后，不直接把事件对象交给 `json-render`。前端需要先在聊天消息投影层中，将该事件转换为稳定的内联表单段模型。该模型再交给下游 UI 容器层与表单协议适配层处理。

这样做有两个好处：

1. 聊天消息结构与表单渲染实现可以解耦。
2. 未来即便更换 `json-render` 或调整 schema 格式，聊天消息流本身的组织方式仍可保持稳定。

### 提交阶段

用户点击提交时，前端先执行本地校验。只有校验通过后，才允许进入消息流。提交成功后，前端需要生成一条新的用户消息，其中至少包含两部分：

1. 面向用户和历史阅读者的可见摘要。
2. 面向运行时与后续 assistant 推理的结构化 JSON 载荷。

这条消息进入当前线程后，后续 run 继续沿既有聊天流程执行。这个设计与当前“用户消息触发 run、run 通过事件流推进界面”的主线保持一致，不额外引入新的历史归档模型。

## 前端分层

当前已确认的前端分层如下：

1. 聊天消息投影层。
2. 聊天 UI 容器层。
3. 表单协议适配层。
4. `json-render` 渲染层。

### 聊天消息投影层

这一层负责消费运行时事件，并把它们归并到聊天消息视图模型中。它需要识别“这是一个表单请求事件”，并在消息流中生成内联表单段，而不是直接操作具体表单控件。

这层的上下文可参考 `frontend-copilot/src/features/copilot/` 下现有的消息流与片段处理模块，例如 `runtime-message-stream.ts`、`run-segment-reducer.ts`、`run-segment-view-model.ts` 与 `CopilotMessageList.tsx` 所代表的职责方向，但本设计不预设这些文件一定已经具备表单能力。

### 聊天 UI 容器层

这一层负责在消息列表中承载内联表单段，处理展示、加载、只读、过期、重试入口等界面状态。它不负责定义协议，也不负责决定 schema 如何解释。

### 表单协议适配层

这一层位于聊天 UI 与 `json-render` 之间，负责将上游给出的规整表单模型转换为 `json-render` 能消费的输入，并把用户填写结果回收成前端统一的提交载荷。它同时负责约束第一版只支持固定动作和基础字段类型。

### `json-render` 渲染层

这一层只做受控 schema 到 React 表单 UI 的映射。它不感知聊天线程、消息历史、run 生命周期，也不决定一条消息何时进入历史。

## `json-render` 的角色与边界

`json-render` 的角色已经明确：它只是受控 schema 的渲染器，不是聊天协议的一部分。

它应承担的职责包括：

1. 根据输入 schema 渲染基础字段类型对应的表单控件。
2. 配合固定动作完成填写、基础校验反馈与提交触发。
3. 在上游要求只读时，按只读状态展示已提交内容。

它不应承担的职责包括：

1. 读取或解释聊天原始消息结构。
2. 维护 thread、run、message 历史。
3. 决定表单是否过期。
4. 处理消息流协议、工具生命周期或运行时状态机。
5. 直接发起聊天消息提交。

因此，`json-render` 的输入必须已经是规整后的表单 schema，而不是运行时原始事件对象，也不是 assistant 原始消息文本。

## 固定动作与交互约束

第一版只支持三个固定动作：填写、校验、提交。

围绕这三个动作，当前交互约束如下：

1. 用户在表单可用期内可以填写字段。
2. 用户提交前可以触发本地校验并看到校验结果。
3. 本地校验失败时，当前操作停留在前端，不进入消息流。
4. 提交成功后，表单立即切换为只读并保留历史展示。
5. 第一版按单次提交设计，不支持同一表单反复提交。

这些约束有助于控制 PoC 复杂度，也能保证聊天历史中每个表单段的语义清晰可追溯。

## 错误处理

### schema 非法

如果运行时给出的 schema 非法，前端应降级为普通 assistant 提示，而不是渲染一个不确定结构的表单。这种降级方式比尝试容错渲染更安全，也更符合受控工具触发的边界。

### 本地校验失败

如果用户填写内容未通过本地校验，提交动作不进入消息流。前端只在当前表单段内反馈错误，保留用户已填写内容，等待修正后再次提交。

### 提交失败

如果提交过程失败，前端应允许重试，并保留当前填写值。这样可以避免用户重复填写，也能让失败恢复维持在同一条聊天上下文内。

### 线程变化或回合失效

如果当前线程发生变化，或者该表单所属回合已经失效，前端应将表单标记为过期。过期状态下不再允许继续提交。

## 兼容性风险

当前已明确的一项现实风险是：`json-render` 与现有前端依赖栈之间存在兼容性验证需求，尤其需要确认它与当前 React 版本及相关依赖组合是否可稳定工作。

这项风险不应混入功能设计本身，而应在 PoC 开始前单独验证。验证结论会直接影响后续接入方式、封装成本与是否需要替代渲染方案。因此，兼容性验证应被视为 PoC 的前置检查项，而不是实现阶段的临时补救动作。

## PoC 范围

当前 PoC 范围严格限定如下：

1. 只支持一个受控工具。
2. 只支持基础字段类型。
3. 只支持单次提交，提交后只读。
4. 提交后只生成一条新的用户消息。
5. 该用户消息包含结构化 JSON 载荷。

PoC 明确不支持以下能力：

1. 流式 patch。
2. 复杂嵌套布局。
3. 文件上传。
4. 动态增删字段。
5. 跨表单依赖。
6. 草稿恢复。
7. secret 或高风险信息采集。

## 验收标准

当下可以用下面这组标准判断设计是否被正确实现：

1. 只有当 AI 调用受控表单工具时，聊天界面才出现内联表单。
2. 表单展示位置位于聊天消息流内部，而不是独立页面或弹窗主路径。
3. 前端表单渲染入口来自结构化事件，而不是 markdown 猜测。
4. `json-render` 只接收规整后的 schema，不接触聊天原始消息结构。
5. 用户本地校验失败时，不会生成新的用户消息。
6. 用户提交成功后，会生成一条新的用户消息，且消息同时包含可见摘要与结构化 JSON 载荷。
7. 已提交表单会锁定为只读，并保留在历史消息中。
8. schema 非法时，界面会降级为普通 assistant 提示。
9. 提交失败后，用户可以重试，且填写值仍然保留。
10. 线程变化或回合失效后，表单会进入过期状态，不能继续提交。

## 后续实现建议

后续进入实现前，建议按下面顺序推进：

1. 先验证 `json-render` 与当前前端依赖栈的兼容性，尤其是 React 版本与构建链路。
2. 再在运行时事件模型中确定受控表单请求的结构化事件挂载位置，优先复用既有 `tool_event` 生命周期或工具结果链路。
3. 然后在前端消息投影层定义稳定的内联表单段视图模型，避免让 UI 层直接耦合运行时原始事件。
4. 随后补齐表单协议适配层，明确 schema 规整规则、提交载荷结构与只读 / 过期状态映射。
5. 最后再接入 `json-render` 作为纯渲染层，并围绕消息投影、提交成功、失败重试与过期场景补测试。

如果后续需要补充实现文档，建议继续与 [聊天运行时](../developers/chat-runtime.md)、[运行时接口 / 事件参考](../reference/runtime-events.md)、[前端实现](../developers/frontend.md) 保持同一事实口径，并把新增协议细节写成独立的事件或前端合同页，而不是散落在组件说明中。
