# 2026-04-11 UI 调试性 / 内部实现文本清理设计

## 文档定位

本文档用于将已确认的 UI 文本清理设计正式落档。范围仅覆盖普通用户可见界面的文本输出清理规则、处理分组、优先顺序、验证口径与交付要求，不包含代码实现，不展开实现计划，也不修改 Markdown 以外的文件。

## 背景与问题

当前普通用户可见界面中，仍存在若干会暴露工程实现细节的文本输出，重点分布在聊天主界面、模型选择器、设置页、运行态空状态与错误提示、thinking 相关展示。这些文本中混入了内部实现、兼容迁移、状态枚举、原因码、调试诊断、架构链路等信息。它们可能对开发排障有帮助，但不应继续暴露给普通用户，否则会造成以下问题：

- 产品反馈与工程噪音混杂，用户难以判断自己真正需要采取的动作。
- 设置说明区与运行态提示承担了超出产品交互范围的内部解释。
- 同一类问题在不同入口以不同内部术语表达，破坏界面一致性。
- thinking 相关展示被工程诊断语义污染，增加普通用户理解成本。

本次清理的目标不是弱化产品反馈，而是去除用户不需要感知的工程实现与诊断噪音，同时保留必要的动作引导。

## 目标与非目标

### 目标

1. 清理普通用户可见界面中的文本输出，重点覆盖聊天主界面、模型选择器、设置页、运行态空状态与错误提示、thinking 相关展示。
2. 从普通 UI 中移除任何暴露内部实现、兼容迁移、状态枚举、原因码、调试诊断、架构链路的信息。
3. 对于用户必须采取动作的提示，仅保留纯用户视角的动作引导文案。
4. 按用户可见路径完成复核，确保普通用户界面中不再出现内部实现、兼容迁移、诊断原因码等信息。
5. 为后续实现阶段定义统一的清查报告口径，要求逐文件记录命中文本、处理动作与保留理由。

### 非目标

1. 本文档不包含代码实现。
2. 本文档不展开实现计划。
3. 本轮不清理非普通用户可见通道中的开发日志、控制台输出或其他内部调试通道。
4. 本轮不削弱必要的产品反馈；凡是用户必须采取动作的场景，仍需保留明确引导。
5. 本轮不改动字段标签、按钮文案、搜索空状态、基础操作反馈等纯产品交互文本，除非其内容已混入内部实现语义。

## 清理判定规则

### 1. 清理对象

清理对象限定为普通用户可见界面中的文本输出，重点覆盖以下界面与链路：

- 聊天主界面
- 模型选择器
- 设置页
- 运行态空状态与错误提示
- thinking 相关展示

判定时以最终用户可见输出为准。若某段文案由辅助函数、状态映射或选项构造逻辑生成，但最终会直接出现在普通 UI 中，仍应纳入本轮清理范围。

### 2. 应移除的文本

以下文本一律判定为应移除对象：

- 暴露内部实现的信息
- 暴露兼容迁移历史的信息
- 暴露状态枚举的信息
- 暴露原因码的信息
- 暴露调试诊断的信息
- 暴露架构链路的信息

判定标准的核心是：即使这些信息对开发排障有帮助，只要普通用户完成当前任务并不需要感知它们，就不应继续出现在普通 UI 中。

### 3. 可保留的文本

只有当用户必须采取动作时，才允许保留提示文本；但保留时必须满足以下条件：

- 文案必须改写为纯用户视角。
- 文案只说明用户需要采取的动作，不解释内部判定过程。
- 文案不得包含工程术语、兼容术语或诊断术语。

禁止在普通 UI 保留以下术语或同类表达：provider catalog、legacy、route ref、adapter、diagnostic、code、stage。

允许保留的动作引导示例仅限于这类用户可执行表述：

- 当前选择已失效，请重新选择。
- 尚未完成配置，请先设置模型服务。

### 4. 可原样保留的文本

以下纯产品交互文本可原样保留：

- 字段标签
- 按钮文案
- 搜索空状态
- 基础操作反馈

前提是其内容本身没有夹带内部实现、兼容迁移或诊断噪音。

### 5. 判定总表

| 判定结果 | 判定标准 | 处理要求 |
| --- | --- | --- |
| 删除 | 文本只承担内部说明、兼容迁移解释、诊断输出、状态枚举、原因码、架构链路说明 | 从普通 UI 中直接移除，不保留替代文案 |
| 改写 | 文本与用户动作存在直接关系，但当前表述混入内部术语或诊断语义 | 改写成纯用户视角动作引导，不解释内部原因 |
| 保留 | 文本属于纯产品交互反馈，且不暴露工程实现信息 | 允许原样保留，但需在清查报告中说明保留理由 |

## 处理分组策略

### A 直接删除

A 组适用于以下内容：

- 纯内部说明
- 纯诊断输出
- 设置页中解释迁移历史的说明
- 设置页中解释内部数据结构的说明

这类文本不承担普通用户操作所必需的信息传达职责，应直接从普通 UI 删除，不改写、不换名保留。

### B 改写后保留

B 组适用于用户必须采取动作的提示。处理原则如下：

- 仅保留动作引导，不保留工程背景解释。
- 仅保留用户下一步该做什么，不保留原因码、阶段名或内部状态标签。
- 若原文案中既有动作引导又有诊断说明，改写后只保留动作引导部分。

本轮已确认的典型场景包括：

- 当前选择已失效，请重新选择。
- 尚未完成配置，请先设置模型服务。

### C 原样保留

C 组适用于纯产品交互文本，包括字段标签、按钮文案、搜索空状态、基础操作反馈等。这类文案的保留前提是：

- 不含内部实现与兼容迁移语义。
- 不含诊断原因码、状态枚举或架构链路信息。
- 不会让普通用户误以为需要理解内部机制才能完成操作。

所有 C 组条目都必须在清查报告中附保留理由，明确说明其属于纯产品交互文本而非工程解释文本。

## 优先级与范围

### 处理优先级

本轮按以下顺序推进并复核：

1. 先处理聊天区与模型选择链路。
2. 再处理设置页说明区。
3. 最后复核 thinking 相关展示。

这一顺序用于确保首先清除用户最频繁接触路径中的工程噪音，其次清理设置页中的说明性冗余，最后收口 thinking 相关展示中的残留内部术语。

## 验证与验收

### 验证方式

验证必须按用户可见路径复核，而不是只按源码位置抽查。至少应覆盖以下路径：

- 聊天主界面中的常态消息区、空状态、错误提示与运行态反馈。
- 模型选择器中的当前选择、失效选择、空结果与动作提示。
- 设置页说明区中的配置提示、失效提示与说明文案。
- thinking 相关展示中的状态说明、空状态与错误提示。

### 验收目标

验收目标是：普通用户界面中不再出现内部实现、兼容迁移、诊断原因码等信息。

只要在普通用户界面中仍能看到工程实现说明、兼容迁移说明、状态枚举、原因码、调试诊断或架构链路信息，本轮清理即视为未完成。

### 修改归类要求

每处命中的文本修改都必须归类为以下三类之一：

- 删除
- 改写
- 保留

其中，所有保留项都必须附理由，明确说明为何该文本属于纯产品交互反馈，且不会向普通用户暴露工程实现与诊断噪音。

### 清查报告要求

后续实现完成后，必须交付一份清查报告。该报告是本轮设计要求的一部分，需逐文件列出命中的冗余文本、处理动作与保留项理由。报告至少应包含以下字段：

| 字段 | 要求 |
| --- | --- |
| 文件 | 逐文件记录命中位置，优先覆盖本文档列出的高优先级文件 |
| 文本位置 | 标明文案所在界面、状态或提示场景，便于按用户路径复核 |
| 原文本 | 记录命中的原始冗余文本 |
| 处理动作 | 明确标记为删除、改写或保留 |
| 改写结果 | 若为改写项，记录最终用户视角文案 |
| 保留理由 | 若为保留项，说明其属于纯产品交互文本且无工程噪音 |

### 最终交付物

本轮整体交付预期包含两项，但本文档只负责明确要求，不负责实现：

1. 代码层面的 UI 清理结果。
2. 一份清查报告，逐文件列出命中的冗余文本、处理动作、保留项理由。

## 风险与回归关注点

### 1. 误删必要动作引导

若把用户必须采取动作的提示与内部诊断说明一并删除，会导致界面反馈不足。控制原则是：动作引导可以保留，但必须改写为纯用户视角。

### 2. 改写后仍残留工程术语

若改写只做字面缩短，但仍保留工程术语或诊断词汇，则会形成伪清理。控制原则是：改写项必须彻底去除内部术语，不得保留 provider catalog、legacy、route ref、adapter、diagnostic、code、stage 等术语或同类表达。

### 3. 过度扩散到纯产品交互文本

若把字段标签、按钮文案、搜索空状态、基础操作反馈一并当作冗余文本处理，会扩大改动面并造成不必要回归。控制原则是：纯产品交互文本应归入 C 组原样保留。

### 4. 设置页说明区与聊天运行态口径不一致

若同类问题在聊天区与设置页分别采用不同动作引导，会造成用户理解成本上升。控制原则是：所有改写项必须统一为用户动作导向，而不是各自保留不同的内部解释残片。

### 5. thinking 相关展示遗漏复核

thinking 相关展示容易夹带工程语义，且分散在状态说明、错误提示与展示文本中。控制原则是：thinking 区域必须作为最后一轮专项复核对象，不能因为前两轮已完成而默认无风险。

## 结论

本轮设计已经明确：清理对象只限普通用户可见界面中的文本输出；任何暴露内部实现、兼容迁移、状态枚举、原因码、调试诊断、架构链路的信息都应从普通 UI 中移除；只有用户必须采取动作时，才允许保留经纯用户视角改写后的提示；字段标签、按钮文案、搜索空状态、基础操作反馈等纯产品交互文本可原样保留，但保留项必须在清查报告中附理由。

后续实现与复核必须以本文档为唯一设计依据，按聊天区与模型选择链路、设置页说明区、thinking 相关展示的优先顺序推进，并按用户可见路径完成验收与逐文件清查报告。
