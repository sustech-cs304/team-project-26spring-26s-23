# 后端 Pydantic 化改造系统设计与执行计划

## 1. 背景与目标

为了提升后端系统数据校验的严谨性、开发体验及可维护性，将对后端进行从 P0 到 P2 的系统性 Pydantic 化改造。

**总体策略**：优先收益，先改高价值的协议边界。采用**单批次、可回退、逐批验收**的执行骨架。

## 2. 非目标

- 本轮**不进行**全仓对象风格重写。
- 内部业务逻辑、现有 dataclass、ORM 实体可暂时保留。
- 第三方抓取解析结果不在最底层直接强套严格模型（将在归一化产物上建模）。
- 暂不运行 live 与 e2e 测试。

## 3. 约束与兼容性

- **协议兼容**：基本保持现有对外协议与字段 shape 不变。
- **字段规范**：默认保持字段名、默认值语义、可空性、嵌套 shape 不变。
- **异常处理**：允许修复少量明显不合理的字段定义。**所有这类修复必须补测试并在本文档底部（“明显不合理字段修复记录位”）单独记录。**
- **扩展性**：对外协议、bridge 协议、history API 默认采用严格字段控制；需要扩展的对象使用显式 metadata/details 等扩展桶。

## 4. 边界建模原则

本轮改造的核心是**边界显式化**：
- **引入范围**：只在边界层优先引入 Pydantic，包括 HTTP 输入输出、bridge 消息、工具参数与结果、history API 返回体、配置文档对象。
- **数据流转**：边界先用 Pydantic 模型接住输入输出，再按需要转换。
- **重点收紧对象**：重点优先收紧联合模型与嵌套子模型，例如 runtime options、thinking、tool 权限、SSE/运行事件 type 变体、bridge 的 operation-specific payload/result、history timeline/block/terminal state 等。
- **迁移顺序**：以“入口校验 + 出口序列化模型化”为先，不追求一步删掉所有旧手写解析。

## 5. 批次计划

以下批次顺序已获批准，须严格按序执行：

- **P0-1**：`backend/app/copilot_runtime/protocol.py`、`backend/app/copilot_runtime/contracts.py` 及直接相关值对象。
  - 目标：把请求/响应/事件的手写解析逐步替换为 Pydantic 模型。
- **P0-2**：`backend/app/desktop_runtime/` 中 capability bridge / host route 等协议对象。
  - 目标：收敛 schema、解析和运行时校验的重复表达。
- **P1-1**：`backend/app/integrations/sustech/` 的工具参数与结果模型。
  - 目标：只触碰 facade / tool boundary，不深入抓取解析深处。
- **P1-2**：`backend/app/copilot_runtime/persistence/query_dtos.py` 及相关 history API 读模型。
  - 目标：让 timeline / block / snapshot 的 shape 显式化。
- **P2**：thinking / provider catalog / 其他配置能力模型。

## 6. 测试与验收标准

- **测试范围**：每一批必须运行全部非 live 自动化测试 + `backend/check.py`。
- **验收机制**：每一批视为独立可验收的小发布。
- **行为验证**：兼容性核对重点为字段名、默认值、可空性、未知字段处理、序列化输出与当前契约一致性。
- **完成条件**：只有 P0 至 P2 全部批次都完成且每批均通过验收，整个项目才算完成。

## 7. 回滚/止损策略

- **隔离原则**：一次只改一个高风险子域，避免交叉污染。
- **故障控制**：一旦某批次测试或验收失败，**必须停留在当前批次内修复**，绝不能把问题带到下一批。
- **行为回退**：若 Pydantic 化导致协议行为偏移，优先保留模型但恢复旧字段语义。
- **问题阻断**：若某批暴露结构性问题，应先记录问题与补丁思路，再继续推进。

## 8. 每批通用执行清单

针对上述每个批次，执行者需遵循以下检查项：
1. [ ] 确认前置批次已全部完成并通过验收。
2. [ ] 梳理当前子域的边界字段，对照旧协议进行建模。
3. [ ] 替换/新增 Pydantic 模型，确保满足“边界建模原则”。
4. [ ] 运行 `backend/check.py` 和全部非 live 自动化测试。
5. [ ] 若发现不可避免的字段逻辑修复，记录至“明显不合理字段修复记录位”并补充相应的测试用例。
6. [ ] 提交当前批次的代码，并进行独立验收。
