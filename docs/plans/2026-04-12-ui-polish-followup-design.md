# 2026-04-12 UI 抛光后续四项设计

## 文档定位

本文档用于将已确认的四个后续 UI 抛光点正式落档。范围仅覆盖错误详情叠加层与聊天工具栏的四项定向调整，不包含代码实现，不展开实现计划，也不修改 Markdown 以外的文件。

## 背景与问题

在上一轮错误详情叠加层与聊天工具栏调整完成后，界面主体方向已经成立，但仍存在四个需要继续收束的 UI 细节：

1. 错误详情叠加层首块与顶部边界线之间的垂直节奏仍不够稳定，打开后第一块内容偏贴近上边界。
2. 原始 `details` 即使本身已经是合法 JSON，当前仍按普通纯文本阅读，结构信息利用不足。
3. 聊天区思考入口仍偏向孤立的小图标按钮，和相邻触发器不在同一组语言里。
4. 工具摘要文案过长，信息密度与视觉噪音偏高，不利于工具栏整体收敛。

本轮仅处理上述四点，目标是在不扩大重构范围的前提下，把现有方案从“可用”进一步收束到“稳定、清晰、一致”。

## 目标与非目标

### 目标

1. 只处理四个已确认的 UI 抛光点，不扩展到无关重构。
2. 明确错误详情叠加层顶部节奏修复的首选设计方向。
3. 明确原始 `details` 在合法 JSON 条件下的结构化阅读展示与复制边界。
4. 明确思考入口的按钮语言、占位策略与工具栏一致性原则。
5. 明确工具摘要文案的收敛规则、提交边界与验证策略。

### 非目标

1. 本文档不包含代码实现。
2. 本文档不展开实现计划。
3. 本轮不改造错误详情叠加层的整体层级结构。
4. 本轮不把 JSON 阅读展示扩展到所有文本块或所有错误字段。
5. 本轮不重新定义整个聊天工具栏的布局系统，只做同一组语言内的收束。
6. 本轮不处理四项之外的额外交互重构、文案体系重写或全局样式翻修。

## 四项修改的设计决定

### 1. 错误详情叠加层首块间距修复

错误详情叠加层顶部节奏问题，优先通过 [`frontend-copilot/src/features/copilot/error-detail-overlay.css`](../../frontend-copilot/src/features/copilot/error-detail-overlay.css) 中 body 顶部 `padding` 或首块专属顶部间距来修复，不调整整体层级结构，也不通过引入额外包裹层来重新分段。

这样约束的原因是：

- 问题本质是顶部留白节奏不足，而不是信息架构错误。
- 用 body 顶部内边距或首块专属间距可以最小范围修正视觉节奏。
- 保持现有层级结构稳定，可以避免把这次小型 UI 抛光演变成结构性重排。

设计判断上，本次应以“让第一块与上边界线之间恢复自然呼吸感”为准，而不是追求新增明显的大空白。若已有块间距节奏合理，则只修正首块与顶部边界之间的特殊关系，不改后续块之间的统一间距规则。

### 2. 原始 `details` 的 JSON 阅读展示

当错误详情中的原始 `details` 可被确认解析为合法 JSON 时，展示层应改为结构化 JSON 阅读模式，而不是继续按普通文本块或代码块平铺。这里应优先复用工具调用展示链路中已存在的 JSON viewer / structured content 模式，例如 [`ToolStructuredContent()`](../../frontend-copilot/src/features/copilot/CopilotMessageList.tsx:667) 所代表的这一类阅读方式。

该决定的关键边界如下：

- 仅对“原始 `details`”这一类内容生效。
- 前提是 `details` 本身是合法 JSON。
- 其他普通文本块、摘要块、说明块保持现有文本阅读语义，不被一并改造成 JSON viewer。
- 若 `details` 不是合法 JSON，则保持现有纯文本代码块或文本块展示，不做强制转换。

JSON viewer 默认展开到第二层，以兼顾两类目标：

1. 初始打开时能直接暴露主要对象层级，减少用户逐层点开的摩擦。
2. 保留更深层嵌套的可折叠性，避免原始诊断信息一次性铺满面板。

这里的设计重点是“提升阅读质量”，不是“改变数据本身”。因此该展示只改变界面中的阅读方式，不改变底层原始字符串的保留语义。

### 3. 思考入口改为框式触发器

聊天区思考入口应在 [`frontend-copilot/src/features/copilot/CopilotComposer.tsx`](../../frontend-copilot/src/features/copilot/CopilotComposer.tsx) 中从纯灯泡图标按钮改为与相邻触发器一致语言的框式按钮。新触发器的基本结构为：左侧灯泡图标，右侧显示当前标签文本，例如“高”“中”“最大”。

这一调整的设计目的不是单纯“把图标旁边加字”，而是将思考入口从孤立工具按钮升级为可被直接阅读的状态触发器。用户无需先理解图标含义，再点开确认当前档位，而是在工具栏静态状态下就能知道当前思考标签。

当没有可用标签时，仍应显示稳定、简短的占位文案，而不是退化为只剩图标。这样可避免以下问题：

- 工具栏内部同类控件出现语义不连续。
- 因数据暂缺导致按钮宽度和视觉重心剧烈波动。
- 用户误以为该入口不可用或缺少当前状态。

占位文案应追求短、稳、可持续，不承担解释性句子职责。它的主要作用是维持按钮作为“带文本状态触发器”的语言一致性。

### 4. 工具摘要文案简化

工具摘要文案改动集中在 [`buildSelectedToolSummary()`](../../frontend-copilot/src/features/copilot/components/ToolPicker.tsx:198) 一类映射逻辑，目标是在不损失可访问性的前提下，将原有“当前已选择 N 项工具”收敛为以下两种状态：

- `未启用工具`
- `启用 N 项工具`

这一收敛方向的设计价值在于：

- 去掉“当前已选择”这类对当前上下文重复解释的铺垫。
- 保留最必要的状态信息，即是否启用以及启用了多少项。
- 让工具摘要与思考入口、附件入口等工具栏触发器更容易形成统一的短句节奏。

文案收敛不意味着牺牲可访问性。若界面上的可见文本已经简化，则可访问名称、提示文本或语义补充仍应保证用户能够理解其作用，但不应把长句重新堆回主视觉层。

## JSON 展示与复制边界

原始 `details` 改为 JSON viewer 后，复制逻辑不随之改变。总复制与单块复制继续输出纯文本，且原始 `details` 的复制必须保留原文本或原始纯文本格式，而不是复制成 viewer 专属结构、折叠状态描述或重新格式化后的界面快照。

边界明确如下：

1. 阅读展示可以结构化，但复制语义仍以原始纯文本为准。
2. 总复制应继续输出面向转交与排障的纯文本汇总，不因为存在 JSON viewer 就混入仅服务于界面的树状阅读标记。
3. 单块复制中的原始 `details` 仍应优先保留其原始文本内容，避免复制结果与上游错误源脱节。
4. 若 `details` 非法 JSON，则展示与复制都沿用现有纯文本路径，不做额外兼容分支扩张。

该边界的核心原则是：界面阅读质量可以升级，但对外转交、排障留痕与复制稳定性不能退化。

## 工具栏一致性原则

本轮不要求聊天工具栏三个触发器达到像素级完全一致，但必须统一到同一组视觉语言。对于思考入口改造，至少需要满足以下一致性原则：

- 与相邻触发器保持相近的高度。
- 使用相近的圆角与内边距节奏。
- 使用相近的文字权重与边框表达。
- 保持同一类“框式触发器”语义，而不是继续维持一个明显更像孤立图标按钮的入口。

这里强调的是“同一组语言”而不是“完全相同组件”。原因在于三个触发器承载的信息密度并不完全一致，允许在宽度或局部布局上存在适度差异；但这些差异不能大到让思考入口继续像额外外挂在工具栏边上的独立小控件。

## 提交与验证策略

### 提交边界

本轮实现阶段必须保持四次独立 git 提交边界，不将四项 UI 抛光混合为单个大提交：

1. 提交 1：叠加层首块间距修复。
2. 提交 2：原始 `details` 的 JSON 阅读展示，且复制保持纯文本。
3. 提交 3：思考入口改为“灯泡 + 当前标签”的框式触发器。
4. 提交 4：工具摘要文案简化。

每次提交都应带上与该项最相关的测试改动，不将四类测试全部堆到最后一个提交中统一补齐。这样做的主要收益是：

- 便于回溯某一项 UI 抛光引入的行为变化。
- 便于在 review 中按主题审阅，而不是在一个混杂提交里拆语义。
- 便于后续必要时按单项回退，而不破坏其他已确认改动。

### 验证顺序

验证策略分为两层：

1. 每项修改完成后，先运行最相关的子集测试，确认该项自身行为稳定。
2. 四项全部完成后，再统一跑一次前端总回归，确认组合后没有产生交叉回归。

该顺序的重点是先做局部收敛，再做总体验证，避免把问题全部拖到最终一次大回归里集中暴露。

## 风险与回归关注点

### 错误详情叠加层节奏风险

若顶部留白修复采用了过重的全局内边距，可能会连带改变所有块的整体可视区密度，导致面板在小尺寸下显得更拥挤。因此应优先针对顶部节奏本身下手，而不是顺手放大全局容器留白。

### JSON 判断边界风险

若对所有文本块都尝试自动 JSON 化，可能会误伤本该按纯文本阅读的内容，造成阅读语义跳变。因此 JSON viewer 仅服务于可确认是合法 JSON 的原始 `details`，不扩大适用面。

### 复制一致性风险

若展示改造后复制路径误接入 viewer 展开结果，可能造成复制文本与原始错误源不一致，影响排障与 issue 转交。实现时必须把“阅读展示层”和“复制输出层”明确分离。

### 工具栏视觉回归风险

思考入口加上标签后，若尺寸、边框或排版节奏处理不当，可能会让工具栏局部显得过重，或者与相邻触发器再次失衡。因此需要以“同组语言、一致节奏”为主要验收口径，而不是只检查文字是否显示出来。

### 文案收敛风险

工具摘要文案简化后，若相关无障碍语义未同步补足，可能让部分用户只看到短句却失去更完整的状态说明。因此可见文案可以变短，但无障碍语义不应被一并削弱。

## 结论

本轮设计严格限定在四个已确认 UI 抛光点内：修正错误详情叠加层首块顶部节奏、仅在原始 `details` 为合法 JSON 时启用结构化阅读展示并保持复制纯文本、将思考入口升级为“灯泡 + 当前标签”的框式触发器、将工具摘要文案收敛为更短的状态表达。

实现阶段应严格遵守四次独立提交边界，并执行“逐项子集验证 + 最终总回归”的验证顺序。本文档仅作为设计定稿，不承担实现计划职责。
