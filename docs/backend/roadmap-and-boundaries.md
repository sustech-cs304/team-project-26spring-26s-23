# 边界与路线图

这篇文档专门用来回答一个在当前仓库里最容易答错的问题：**哪些东西已经可以当成当前能力来依赖，哪些只是代码里能调用，哪些还只是未来草案。**

如果不把这三层分开，读者很容易因为目录名、依赖名或函数名产生过度乐观的判断，最后把文档写成“系统好像已经服务化完成”，而这和代码现状并不一致。

## 先给总体判断

当前 `backend/` 更像：

- 一套围绕 Blackboard 与 TIS 的 Python 能力库；
- 带有少量明确 CLI 入口；
- 已具备数据同步与本地持久化能力；
- 但还没有形成统一、稳定、面向外部系统的 Web API 服务。

因此，当前文档中的第一优先级，不是讲“未来可以怎么扩展”，而是先把真实边界说清楚。

## 一、已实现

### 1. Blackboard 有明确运行面

这是当前最可以放心写进“现状”的部分。

已经明确存在的事实包括：

- 课程目录搜索 CLI；
- 日历 ICS 同步 CLI；
- provider use case，把抓取过程组装成更完整的调用流程；
- data 层，把结果同步到本地 SQLite；
- 覆盖 provider / data / API 解析的测试；
- live 集成测试与 e2e 测试，用于真实链路验证。

更直白地说，**Blackboard 现在不只是“代码能跑几段”，而是已经形成了一条比较完整的抓取、整理、同步、输出链路。**

### 2. 当前输出是可观察的

虽然不是 HTTP API，但当前已有几类现实可观察输出：

- CLI 终端日志与预览；
- CLI 生成的 JSON 报告；
- 工具函数返回的字典；
- use case 返回的结构化结果对象；
- 数据库中的同步结果与统计。

这些东西已经足够支撑“当前系统确实有能力、有边界、有输出”。

### 3. TIS 并不是空壳

TIS 侧已经存在：

- 链路诊断；
- 个人成绩抓取；
- 学分绩抓取；
- 已选课程抓取；
- 可选持久化；
- 对应单元测试与 live 集成测试。

所以文档不能把 TIS 写成“还没开始做”。真实情况是：**它已经有可调用能力，但外层入口和产品化形态还不如 Blackboard 清楚。**

## 二、代码里可调用，但不是正式入口

这是最容易被误写的一层。

### 1. Blackboard 工具层函数

当前 Blackboard 的工具层函数会返回字典，并且有测试校验这些返回结构。但它们的正确定位是：

- 已实现；
- 可在 Python 代码中直接调用；
- 当前可作为可观察输出；
- 不是正式 HTTP 接口。

因此，不应把这些函数写成“已开放给前端的 API”。更准确的说法是：**它们是当前后端内部/邻接层可调用工具。**

### 2. Blackboard snapshot 同步 use case

`snapshot_sync` 相关能力已经很完整：会抓 Blackboard 课程及下游数据、构建 payload、同步数据库、计算预期活跃数，还可执行第二次同步验证。

但这仍然是 provider use case。也就是说：

- 它已经能完成完整任务；
- 但它不是一个用户直接访问的服务端接口；
- 当前更像内部主流程能力。

### 3. TIS provider use cases

TIS 的诊断、个人成绩、学分绩、已选课程 use case，都属于这类状态：

- 代码中可调用；
- 有相对清楚的结果结构；
- 某些场景还能持久化；
- 但缺少面向第一次接手者的正式运行入口。

因此文档里最合适的写法不是“已提供 TIS 接口服务”，而是“已实现若干 TIS provider 能力，可供开发阶段调用与扩展”。

### 4. `api/` 目录本身

当前 `app/blackboard/api/` 和 `app/teaching_information_system/api/` 的意义，是访问上游系统并解析数据。

所以即便这里已经有很多代码，也不能直接写成“系统已经有前端 HTTP API 层”。这会把“调用外部系统的客户端代码”和“向外提供服务的服务端接口”混为一谈。

## 三、未来草案

这一层必须明确标注为草案，不能和现状混写。

### 1. 前后端 HTTP 连接方式

如果未来要给前端提供稳定接口，比较合理的方向，是把现有 provider/use case 与工具层整理成更稳定的资源接口，再补上服务启动、鉴权、错误处理和契约约束。

但截至当前代码事实，这件事还没有形成仓库内可直接运行的正式实现。因此：

- 可以写方向；
- 可以写字段草案；
- 可以写资源边界建议；
- 不能写成“现有 API 规范”或“已承诺接口”。

### 2. 更统一的服务层编排

从目录命名看，未来确实可能出现更明确的服务编排层或应用层收口点。但当前 `app/services/` 还只是一个非常薄的 package，不能提前把这条路线写成“现在已经存在”。

### 3. TIS 的外层运行入口

TIS 未来也完全可能补出 CLI、工具层或服务化封装，但目前这还属于扩展方向，而不是现状说明。

## 当前最需要避免的四种误写

### 1. 把依赖表当成功能清单

`pyproject.toml` 里出现 `fastapi` 和 `uvicorn`，不代表仓库当前已经是 Web API 服务。文档里必须避免这种推断式写法。

### 2. 把目录名当成熟度证明

`api/`、`services/`、`provider/` 这些目录名都有很强的架构暗示，但现状描述必须以代码实际入口和调用方式为准，而不是以目录名脑补完整架构。

### 3. 把工具函数写成 HTTP 接口

当前 Blackboard 工具函数的现实形态是“返回字典的 Python 工具层”。如果把它们写成 `/api/...` 形式的现成接口，就会直接误导使用者。

### 4. 把前后端对接说明写成既成事实

和前端协作时，确实需要描述“以后更可能怎么接”。但这种说明必须明确挂在“草案”标签下，不能包装成今天就能依赖的契约。

## 当前可依赖的现实边界

如果用一句更实际的话来概括：

- **可以依赖** Blackboard CLI、provider、同步/持久化链路，以及它们当前产生的 JSON/字典输出；
- **可以用于开发扩展** TIS provider use cases 和 Blackboard/TIS data 层；
- **不能假定已经存在** 面向前端的稳定 HTTP API 服务。

## 路线图只保留到必要程度

本轮文档不打算替后端提前写完整未来规划，但从现状往后看，比较自然的路线大致只有三条：

1. 继续把 Blackboard 现有 CLI / provider / data 链路打磨得更稳定；
2. 给 TIS 补更明确的外层入口与运行文档；
3. 在未来需要前后端联调时，再把当前工具层与 use case 收束成正式服务接口。

这三条都只是合理方向，不是当前承诺。
