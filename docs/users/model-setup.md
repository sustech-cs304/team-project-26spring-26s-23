# 模型与 Provider 设置

- 这页给谁看：准备配置模型服务、修改默认模型或排查“为什么聊天里没有可选模型”的普通使用者。
- 这页解决什么问题：只从用户动作出发，说明你在设置里要做什么、配置完成后会看到什么、怎样判断已经可用。
- 当前覆盖到哪：覆盖设置工作区里的模型服务配置、默认模型选择和聊天区验证；更底层的 catalog、profile、route 语义收口到共享事实页。
- 当前状态：模型服务配置与默认模型主线已可用；跨 Provider 组合的一致体验部分接通。

先说结论：当前模型设置不是“先切一个全局服务商就结束”。更准确的做法是：新增一条你自己的 Provider 配置，填好密钥和模型列表，再把一条默认模型路由设给聊天区使用。

## 先分清你在界面里实际会碰到的三件东西

| 你在界面里看到的对象 | 用使用者视角怎么理解 | 什么时候需要进一步看共享事实 |
| --- | --- | --- |
| Provider 类型 | 系统认识的服务类型，例如某类 API 兼容服务。 | 想知道目录和运行边界时，看[Provider 与模型路由说明](../reference/providers-and-routing.md)。 |
| 你的模型服务配置 | 你自己保存的一条可用服务，包括地址、密钥和模型列表。 | 想知道为什么它不是“全局 active provider”时，看[Provider 与模型路由说明](../reference/providers-and-routing.md)。 |
| 聊天里实际选中的模型 | 这次聊天真正要走的那条模型路由。 | 想知道默认值和请求级路由怎样分工时，看[Provider 与模型路由说明](../reference/providers-and-routing.md)。 |

## 最短设置流程

### 第一步：进入“模型服务”

在设置工作区里进入“模型服务”。当前这里已经能完成这些动作：

- 新增一条服务配置。
- 编辑名称、地址和密钥。
- 维护模型列表。
- 复制、删除或调整顺序。

### 第二步：填好一条真正可用的服务配置

至少补齐下面这组信息：

- 你要使用的 Provider 类型。
- 可访问的地址。
- 有效的密钥或等价凭据。
- 至少一个可调用模型。

当前最容易误判的一点是：**能新建一条配置，不等于它已经可执行。** 如果地址、密钥或模型列表不完整，聊天区仍然可能没有可用模型。

### 第三步：确认模型列表不是空的

聊天区能不能出现可选模型，取决于你保存的服务配置里有没有真实模型列表。当前系统不会默认替你补出一组示例模型。

### 第四步：进入“默认模型”

如果你希望聊天区一打开就更容易有默认选择，再去“默认模型”里设置主助手模型。这里保存的是默认值，不代表每一次聊天都只能走同一条路由。

### 第五步：回到聊天区验证

回到助手工作区后，先看三件事：

1. 模型选择区是否不再为空。
2. 你刚才设置的默认模型是否能作为初始选择出现。
3. 发送一条简单消息后，是否能得到正常回复。

## 怎样判断当前设置已经真正可用

| 判断点 | 看到什么更像成功 |
| --- | --- |
| 设置页 | 这条配置能正常保存，且模型列表不是空的。 |
| 聊天区 | 模型选择器里能看到你刚保存的模型。 |
| 发送测试 | 回复能正常流式出现，而不是始终失败或停在空状态。 |
| Thinking | 只有在当前模型与 Provider 组合支持时，才会出现相应选择或展示。 |

## 当前最容易写错的几件事

### 不是“切一个 active provider”就结束

当前更准确的理解是：你维护的是自己的 Provider 配置条目，聊天真正使用哪条模型路由，还会看默认模型和本次聊天的实际选择。

### 目录里有某个 Provider，不等于现在就一定能执行

有些条目可以显示在目录里，但真实执行能力要看当前配置和当前支持状态。要统一这层事实，请看[Provider 与模型路由说明](../reference/providers-and-routing.md)。

### Thinking 不是所有模型都一样

你配置好模型之后，也不能默认所有组合都会给出相同的 Thinking 档位、预算或展示效果。这个判断始终取决于模型与 Provider 的组合。完整口径见[Thinking 能力说明](../reference/thinking.md)。
