# 当前限制与能力边界

- 这页给谁看：想先保守判断 CanDue 当前覆盖到哪、哪些部分可以直接试、哪些部分还不能过度期待的普通使用者。
- 这页解决什么问题：把使用者最关心的能力边界收成一页，用统一状态口径帮助你快速做判断。
- 当前覆盖到哪：覆盖桌面工作台、模型设置、聊天、Thinking、外围工作区和若干领域入口；更完整的全站边界收口在共享事实页。
- 当前状态：页面已可用。

先说结论：如果你当前主要看重的是“桌面工作台 + 模型配置 + 助手聊天”这条主线，CanDue 已经值得直接上手。如果你期待的是“所有外围能力都已经成熟接通”，那就需要按部分接通或规划中来判断。

## 已可用

| 主题 | 现在可以怎样理解 |
| --- | --- |
| 桌面宿主上的助手工作台 | 当前主形态已经成立，能进入助手与设置工作区。 |
| 模型服务配置 | 可以新增自己的 Provider 配置、保存密钥和模型列表。 |
| 默认模型设置 | 可以为聊天区提供默认模型路由。 |
| 聊天主线 | 可以选择模型、发送消息、接收流式回复。 |
| Thinking 基础能力 | 在支持的模型与 Provider 组合上，可以选择或观察 Thinking 行为。 |
| 本地聊天历史恢复 | 重启应用或 runtime 后，线程列表、历史时间线和基础 run replay 可以从本地持久化数据恢复。 |

## 部分接通

| 主题 | 为什么要保守理解 |
| --- | --- |
| Thinking 的统一体验 | 不同模型与 Provider 组合的档位、预算和展示效果并不完全一致。 |
| 可见 reasoning 展示 | 有些组合可以显示更多，有些组合不会完整展示。 |
| 历史线程继续对话 | 如果历史模型、工具或 Thinking 能力与当前配置发生漂移，需要先显式重新绑定后再继续发送。 |
| 备份与恢复 | 聊天历史支持本地 SQLite 备份 / 恢复，但它仍然是单机运维能力，不等于云同步。 |
| Blackboard 与 TIS 相关区域 | 已经有真实能力、设置或入口，但还不应当成完整成熟的产品面。 |
| 外部源与 WakeUP 等流程 | 已经有页面和局部交互，但还不能直接写成稳定闭环。 |
| 能力、文件、开发等工作区 | 页面结构已经有了，但真实数据面和业务闭环仍不完整。 |

## 规划中

| 主题 | 当前应该怎样看 |
| --- | --- |
| 跨设备 / 云端同步的完整历史恢复 | 当前不应把本地持久化误解成多设备同步或云端历史。 |
| 围绕 Blackboard、TIS、外部源的完整产品化体验 | 现在不应写成已经整体落地。 |
| 所有工作区都形成稳定成熟的数据面 | 现在还不能按这个标准理解。 |

## 使用时最容易误判的地方

### 目录里有入口，不等于完整能力已经成熟

你在界面里看到某个工作区、某个设置分区、某个领域入口，并不自动等于它已经完成端到端闭环。

### 能配置，不等于所有组合都能给出同样结果

模型能配置、也能聊天，不代表每一种模型与 Provider 组合都会给出同样的 Thinking、reasoning 或工具体验。

### 某个页面能保存字段，不等于下游都已经稳定生效

当前一些设置已经能稳定保存，但它们并不都等于已经成为成熟运行时功能。

### 本地历史恢复，不等于云同步

当前恢复能力依赖本地持久化数据库。它能帮你跨重启找回历史线程，但不代表这些历史会自动同步到其他设备。

## 如果你只想快速判断“现在适不适合我”

### 更适合立即试用的情况

- 你想要的是桌面宿主上的智能助手工作台。
- 你愿意自己配置模型服务。
- 你主要使用聊天主线，而不是期待外围能力全部成熟。

### 需要更保守的情况

- 你期待所有外围工作区都已经和后端深度接通。
- 你期待 Blackboard、TIS、外部源都已经形成完整稳定产品面。
- 你期待所有模型都提供统一的 Thinking 与 reasoning 体验。
