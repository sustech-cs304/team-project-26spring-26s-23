# 未来后端 API 讨论草案

这页属于 frontend 分册的参考/边界层，只用于讨论未来可能需要补齐的接口主题。当前真实实现，请以前面的[前端现在怎样连接后端](./backend-connection-contract.md)、[前端当前 UI 状态](./ui-current-state.md)和[聊天运行时契约](../system/chat-runtime-contract.md)为准。

## 当前已经是事实的部分

下面这些内容已经是当前实现，不属于草案范围：

- 前端已经能通过 hosted runtime 或开发态 override 判断可用连接地址。
- 当前聊天主路径已经围绕 `agents/list`、`session/create`、`capabilities/get` 和 `message/send` 组织。
- 聊天发送区已经在请求里显式携带本次模型和工具选择。
- settings workspace 已经可以保存 provider profiles、默认模型路由和大量设置字段。

因此，这页不应该重写当前主路径，也不提供现成的联调依据。

## 这页应该怎样使用

- 这里只列未来接口主题，不列成已实现接口。
- 这里不承诺固定 HTTP 路径、请求 schema、认证方案或错误码。
- 这里出现的“可能需要”不代表已经排期。
- 需要判断当前实现时，应回到事实层文档和系统层契约文档。

| --- | --- | --- |

| 开发工作台数据面 | `开发` 工作区 | 当前只有结构化页面，还没有真实任务、构建和日志接口。 |

## 几个最值得继续讨论的主题

### 1. 会话恢复与历史回放

这是当前聊天主路径之后最自然的一步。现在已经有会话创建、切换和消息发送，但前端仍然缺少一套更完整的“重新打开后继续聊”的能力，因此未来很可能需要：

- 服务端会话列表。
- 按 `sessionId` 拉取历史消息。
- 会话摘要、更新时间与排序信息。
- 历史消息与当前消息流的合并策略。

### 2. 模型服务和默认模型路由的后端化边界

当前前端已经能保存 provider profiles 和默认模型路由，也会用它们组织聊天模型目录和默认选择。

但如果后续要让这些配置真正进入更完整的后端协同链路，仍然需要单独讨论：

- 哪些字段只是前端设置资产。
- 哪些字段需要形成后端可消费的稳定配置。
- 怎样区分宿主默认模型、工作区默认路由和请求级模型选择。

### 3. WakeUP 与外部源

`外部源` 页面已经有真实入口，但它当前仍然主要停留在前端保存链接和弹窗反馈这一级。若后续要做成熟的外部源接入，通常还需要继续讨论：

- 分享链接解析接口。
- 冲突处理策略。
- 同步结果的数据结构。
- 与课程、资料或日程等领域数据的映射关系。

### 4. 其余三个工作区的数据面

`能力`、`文件` 和 `开发` 三个工作区已经有稳定结构，因此它们未来不是“要不要有页面”的问题，而是“何时补真实数据面”的问题。后续如果推进，会更需要围绕目录、状态、任务、日志和权限等主题定义接口。

## 当前不应该从这页得出的结论

- 不能把这页当作当前接口说明书。
- 不能把这页里的主题写成已经实现。
- 不能把这页里的草案表格当成联调契约。
- 不能因为设置页字段已经存在，就推断后端已经有对应完整接口。
