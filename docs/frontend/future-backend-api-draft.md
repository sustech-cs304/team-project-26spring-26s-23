# 未来后端 API 讨论草案

> **状态说明：草案、未实现、仅供设计讨论。**
>
> 本文不是当前接口文档，不是联调依据，也不是已经确认的实现承诺。它只根据当前前端界面结构、字段分区和后续规划方向，整理“未来可能需要讨论哪些接口主题”。

## 文档用途

这份附录用于回答：**如果后续要把当前前端界面逐步接成真实能力，未来后端侧可能需要讨论哪些配置分组和接口主题。**

它适合在以下场景使用：

- 进行前后端设计讨论时，用来列主题而不是定协议
- 审查当前前端界面时，判断哪些配置分区未来可能需要真实接口支撑
- 为后续接口规划做信息整理，但不提前承诺细节

## 使用边界

- 本文只做“主题整理”，不写成已实现接口。
- 本文不提供固定 HTTP 路径、请求体、响应体、认证头或错误码。
- 本文不能覆盖当前事实边界；当前事实仍以 [backend-connection-contract.md](./backend-connection-contract.md) 和附录现状表为准。
- 本文中的“可能需要”表示设计讨论方向，不表示已经决定必须这样做。

## 必要说明

### 1. 为什么这里只写“主题”，不写接口细节

因为当前前端代码并没有形成足够稳定的后端接口事实。现在如果直接写出：

- 固定 HTTP 路径
- 固定请求 / 响应 schema
- 固定认证流程
- 固定错误码

就会把“讨论材料”误写成“实现事实”。这正是本次文档重构要避免的。

### 2. 为什么 `API 服务器` 相关内容仍然只能写成草案

虽然当前设置页已经有“后端地址”“重连策略”“健康检查轮询”等字段，但这只能说明前端界面已经表达了未来可能需要的配置方向。

它不能说明：

- 已经存在真实接口
- 已经有健康检查协议
- 已经有自动重连实现
- 已经有可联调的请求约定

因此，这部分只能以“未来讨论主题”的方式保留。

### 3. 为什么 `assistant` 相关内容也只能写成草案扩展

当前 `assistant` 工作区虽然最接近主路径，但它现在真正生效的仍只是最小连接字段和状态驱动骨架。

像动态助手列表、真实会话历史、消息流、会话存储这些能力，都还没有在代码里形成已实现事实，所以这里只能整理成未来可能讨论的主题。

## 当前已实现 / 未实现说明

### 当前已实现

- 前端已经暴露出多个未来可能需要真实后端支撑的页面分区和配置分组
- 当前已经能明确最小连接主路径围绕 `runtimeUrl` 和 `agentName` 两个字段展开
- 当前已经能基于前端界面组织，初步归纳未来接口讨论主题

### 当前未实现

- 本文列出的所有未来接口主题本身
- 这些主题对应的固定 HTTP API 设计
- 这些主题对应的请求 / 响应 schema
- 这些主题对应的认证和鉴权流程
- 这些主题对应的稳定联调协议
