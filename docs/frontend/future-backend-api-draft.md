# 未来后端 API 讨论草案

> **状态说明：草案、未实现、仅供设计讨论。**
>
> 本文不是当前接口文档，也不是联调依据。它只整理：在当前前端已经进入 session-first 主路径之后，未来还可能需要继续补哪些接口主题。

## 先确认边界

当前已经是事实的内容包括：

- 前端已经能连接本地 desktop runtime。
- 当前正式聊天主路径已经是：
  - `agents/list`
  - `session/create`
  - `capabilities/get`
  - `message/send`
- 设置页中已有少量正式配置字段进入统一配置中心。

因此，这份草案**不是**在重新设计当前聊天主路径；它更适合回答：

- 还有哪些页面未来需要更完整的后端支撑。
- 现有主路径之外，还缺哪些能力层接口。

## 使用边界

- 本文只列“主题”，不写成已实现接口。
- 本文不提供固定 HTTP 路径、请求体、认证头、错误码承诺。
- 当前事实仍以前端正式文档和系统契约文档为准。
- 文中“未来可能需要”不代表已经排期实现。

## 逐类说明

### 1. 为什么“会话列表持久化与恢复”仍是未来主题

当前系统已经有会话创建和会话切换，但：

- 会话列表主要存在于当前 renderer 窗口内存里
- 前端刷新后不会自动恢复
- 当前也没有服务端提供的“历史会话列表”正式接口

因此这里未来最可能需要讨论的，不是“如何再造一个会话创建接口”，而是：

- 是否需要列出历史会话
- 是否需要最近更新时间、摘要、排序
- 是否需要恢复某个已有会话

### 2. 为什么“历史消息回放”仍是未来主题

当前后端已经会把成功的 user / assistant 消息追加到内存态 session store；但前端当前并不会在切换会话后自动拉完整历史回来。

因此未来如果要做完整聊天体验，可能需要补：

- 按 `sessionId` 读取历史消息
- 分页或增量回放
- 历史消息与当前消息流的合并策略

### 3. 为什么“模型服务”“默认模型路由”仍然是草案主题

当前需要把两类模型语义分开：

- `backendExposed.model`：已经是正式配置字段，用于下一次 runtime 启动时的默认模型投影
- 聊天面板中的模型选择器：当前用于 `message/send` 的请求级模型选择

真正还没有成型的，是设置页中这些更完整的能力：

- 服务商管理
- 模型清单管理
- 任务级模型路由
- 回退模型策略

因此这些内容现在仍然只能保留为未来讨论主题。

### 4. 为什么“API 服务器扩展配置”仍然只是草案

当前已经正式接入的只有：

- 开发态 `runtimeUrl`
- 根装配层状态摘要
- 根层 retry 动作

而同页里这些内容仍未形成正式契约：

- 后端地址
- 重连策略
- 健康检查轮询

因此未来如果真要把这一页补成完整配置页，仍需要单独讨论接口和状态模型。

### 5. 为什么 `capabilities` / `files` / `developer` 仍然主要是未来主题

因为这三页当前共同特点很明确：

- 有入口
- 有结构
- 但没有真实数据回流

所以现在更合适的做法，是把它们当成“未来接口主题的收集器”，而不是误写成“当前已经有一套隐藏 API 只差文档”。

## 当前已实现 / 未实现说明

### 当前已实现

- 前端已经暴露出多类未来可能需要真实后端支撑的页面分区
- 当前聊天主路径已经围绕后端目录、会话绑定和请求级模型 / 工具策略组织
- 当前已经可以据此整理未来接口讨论主题

### 当前未实现

- 本文列出的未来主题本身
- 这些主题对应的固定 HTTP 路径设计
- 这些主题对应的请求 / 响应 schema
- 这些主题对应的认证和错误码方案
- 这些主题对应的稳定联调协议

## 一句话总结

当前前端已经把聊天主路径做实，但围绕历史会话、页面扩展配置、文件工作区、能力中心和开发工作台的数据面，未来仍然需要进一步讨论正式后端接口主题。
