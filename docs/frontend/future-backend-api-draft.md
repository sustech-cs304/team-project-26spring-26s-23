# 未来后端 API 讨论草案（旧资料）

旧的 `frontend` 分册，只讨论未来可能需要补齐的接口主题。真实实现请以[给开发者](../developers/getting-started.md)、[聊天运行时](../developers/chat-runtime.md)、[运行时接口 / 事件参考](../reference/runtime-events.md)和[前端实现](../developers/frontend.md)为准。

## 已是事实的部分

下面这些内容已是实现，不属于草案范围：

- 前端已能通过 hosted runtime 或开发态 override 判断可用连接地址。
- 聊天主路径已围绕 `agents/list`、`thread/create`、`thread/get`、`run/start`、`run/stream` 和 `run/cancel` 组织；`session/create`、`capabilities/get` 和 `message/send` 只是兼容壳。
- 聊天发送区已在请求里显式携带模型路由、Thinking 和工具选择。
- settings workspace 已可保存 provider profiles、默认模型路由和大量设置字段。

这页不重写主路径，也不提供联调依据。

## 这页用法

- 只列未来接口主题，不列成已实现接口。
- 不承诺固定 HTTP 路径、请求 schema、认证方案或错误码。
- 这里出现的"可能需要"不代表已排期。
- 需要判断实现时，回到事实层文档和系统层契约文档。

## 几个最值得继续讨论的主题

### 1. 会话恢复与历史回放

这是聊天主路径之后最自然的一步。已有会话创建、切换和消息发送，但前端缺少一套"重新打开后继续聊"的能力。未来可能需要：

- 服务端会话列表。
- 按 `sessionId` 拉取历史消息。
- 会话摘要、更新时间与排序信息。
- 历史消息与消息流的合并策略。

### 2. 模型服务和默认模型路由的后端化边界

前端已能保存 provider profiles 和默认模型路由，也会用它们组织聊天模型目录和默认选择。

如果后续要让这些配置进入后端协同链路，仍需单独讨论：

- 哪些字段只是前端设置资产。
- 哪些字段需要形成后端可消费的稳定配置。
- 怎样区分宿主默认模型、工作区默认路由和请求级模型选择。

### 3. WakeUP 与外部源

`外部源` 页面已有真实入口，但主要停留在前端保存链接和弹窗反馈。后续若有成熟外部源接入，通常需讨论：

- 分享链接解析接口。
- 冲突处理策略。
- 同步结果的数据结构。
- 与课程、资料或日程等领域数据的映射关系。

### 4. 其余三个工作区的数据面

`能力`、`文件` 和 `开发` 三个工作区已有稳定结构，各自已在不同程度上接入了真实数据面（能力工作区已具备 MCP 管理、技能面板和工具权限；文件工作区已具备文件树 CRUD 操作；开发工作区已具备日历看板、Gantt 和 Timeline 视图），不是"要不要有页面"的问题，而是"何时补完整后端数据面与业务闭环"的问题。后续推进需要围绕目录、状态、任务、日志和权限等主题定义接口。

## 不应从这页得出的结论

- 不能把这页当作接口说明书。
- 不能把这页里的主题写成已实现。
- 不能把这页里的草案表格当成联调契约。
- 不能因为设置页字段已存在，就推断后端已有对应完整接口。
