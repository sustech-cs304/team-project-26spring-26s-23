# 已实现、占位与下一步（旧资料）

这页属于旧的 `frontend` 分册。当前实现事实请优先以[给开发者](../developers/getting-started.md)、[前端实现](../developers/frontend.md)、[能力边界 / 状态总表](../reference/capabilities.md)和[运行时接口 / 事件参考](../reference/runtime-events.md)为准。这里保留的是历史边界说明：哪些内容已经能稳定写成当前实现，哪些内容仍然只是半接通，哪些内容还只是结构占位。

## 当前可以稳定写成已实现的部分

下面这些内容已经不再只是设计草图。

### 1. 桌面前端整体形态已经成立

当前已经稳定存在 Electron 主进程、preload 暴露层和 React renderer 工作台。左侧入口栏、多工作区结构、根装配状态页和设置工作区都已经进入长期演进状态。

### 2. 前端持久化已经明确分成两层

当前已经可以明确写成下面这件事：

- 公开配置中心负责少量公开字段。
- settings workspace 负责设置工作区的大部分普通字段和 secrets。

这意味着设置页已经不是单纯的本地临时交互。很多字段能够保存并在重新打开应用后恢复。

### 3. 聊天主路径已经是 `thread/run`

当前前端已经不是围绕旧全局 agent 入口组织聊天。更准确的路径是：

1. 先确认有可用 `runtimeUrl`。
2. 拉取后端智能体目录。
3. 选择智能体并创建 `thread`。
4. 读取当前 `thread` 的能力面。
5. 在 `run/start` 中显式带上本次模型路由、Thinking 和工具选择。
6. 通过 `run/stream` 消费事件流；需要时再走 `run/cancel`。

`session/create`、`capabilities/get` 和 `message/send` 仍然保留，但它们当前是兼容壳。

### 4. 聊天发送区已经有真实请求语义

模型选择器和工具选择器都已经进入真实发送链路。它们不只是界面装饰。

同时也要保持另一条边界：聊天发送区使用的模型目录，来自 settings workspace 里的 provider profiles 与默认模型路由，而不是前端静态内置默认模型。

## 当前处于半接通状态的部分

下面这些内容已经有入口、字段或局部交互，但还不适合写成成熟闭环能力。

| 主题 | 当前已经有的部分 | 当前边界 |
| --- | --- | --- |
| `SUSTech 信息` | 页面、字段保存、CAS 密码保存、Blackboard 偏好输入 | 下游业务接线仍然有限，`TIS 信息` 也仍保留扩展位。 |
| `模型服务` | provider profiles、模型清单、API 密钥、复制删除排序等交互 | 页面很成熟，但它并不等于整套后端模型管理能力已经完成。 |
| `默认模型` | 主助手模型和快速执行模型路由可保存 | 它会影响聊天草稿默认值，但不等于每次消息已经完全由这页统一控制。 |
| `API 服务器` | 开发态 `runtimeUrl`、地址草稿、重连策略、轮询字段 | 只有 `runtimeUrl` 直接进入当前连接主路径，其余字段仍然更多是设置资产。 |
| `外部源` / WakeUP | 分享链接可保存，前端可触发解析弹窗 | 当前仍主要是前端半接通流程，不能写成成熟的后端同步能力。 |
| 会话列表 | 当前窗口内可以创建、切换、排序与管理会话 | 跨刷新恢复、服务端历史目录和完整回放仍未完成。 |

## 当前仍然主要是结构占位的部分

下面这些页面已经有稳定位置和基本布局，但当前还不适合写成成熟业务页：

- `能力` 工作区。
- `文件` 工作区。
- `开发` 工作区。

更准确的描述是：这些页面的 Hub 结构已经稳定，未来要承载的主题也已经可见，但真实数据面、业务联动和后端回流还没有补齐。

## 当前最容易被夸大的几件事

- 设置页字段很多都能保存，不代表它们都已经接进 runtime 或业务真源。
- `backendExposed.model` 仍然存在，不代表它就是聊天面板每条消息最终使用的模型。
- `外部源` 页面已经出现，不代表 WakeUP 已经打通成熟后端接入。
- `能力`、`文件`、`开发` 三个工作区已经有页面，不代表它们已经拥有成熟数据面。
- 会话列已经可用，不代表完整会话恢复和历史回放已经完成。

## 下一步更适合作为什么理解

这页标题里保留“下一步”，但这里记录的不是排期承诺，而是当前最清楚的后续方向：

- 继续补齐会话恢复、历史消息回放和更完整的会话产品能力。
- 继续让 settings workspace 里真正需要长期生效的字段接入明确 owner。
- 继续把 WakeUP、外部源和其余工作区从结构入口逐步推进到真实数据面。
