# 已实现、占位与下一步

这篇文档不讲细节实现，而是帮读者快速判断一件事：

> 当前前端到底哪些已经是稳定事实，哪些只是局部接通，哪些还只是页面结构。

如果你准备继续开发，先把这三类边界分清，会比直接盯着页面外观更有用。

## 先给结论

当前前端已经不再只是桌面壳或页面草图，而是有三条明确主线：

1. **统一配置中心已经正式落地**。
2. **聊天主路径已经切到 session-first**。
3. **工作台 UI 已经完成一轮现代化收敛**。

但这并不代表：

- 设置页所有字段都已经接好；
- 聊天已经具备完整历史回放和会话恢复；
- 其余工作区都已经有真实业务数据。

## 第一类：当前可以放心写成已实现的部分

### 1. Electron 桌面工作台已经稳定存在

当前已经明确存在：

- Electron 主进程
- preload 预加载层
- React renderer 工作台
- 左侧图标栏与多工作区结构
- 根级启动壳与错误兜底

这说明当前前端已经是可以持续演进的桌面应用，不再只是页面原型。

### 2. 统一配置中心已经是一条正式主链路

当前统一配置中心已经具备：

- 分域多文件 JSON
- 可读磁盘文件
- 主进程归一化
- legacy migration
- 公共快照 / 公共补丁 / 更新订阅

当前正式字段包括：

- `theme`
- `animationsEnabled`
- `agentName`
- `runtimeUrl`
- `model`

其中：

- `theme`、`animationsEnabled` 会立即影响前端显示
- `runtimeUrl` 是开发态 override 字段
- `model` 会影响下一次后端启动时的默认模型投影
- `agentName` 仍然持久化，但已不再是聊天 readiness 的硬门槛

### 3. 助手工作区已经进入 session-first 聊天主路径

当前聊天主路径已经可以写成：

1. 先确认有可用 runtime URL
2. 拉后端智能体目录
3. 选择智能体
4. 创建会话
5. 拉取会话能力面
6. 发送消息，并在请求里显式带上模型和工具选择

这意味着当前前端已经不再靠“全局 agent + 旧聊天桥”组织主入口。

### 4. 当前聊天 UI 已经有一套稳定外观

当前聊天面板已经具备：

- 状态分层清楚的启动 / 错误 / 就绪提示
- 三栏助手工作区布局
- 可滚动消息区
- 底部输入区
- 发送按钮
- 模型选择器
- 工具选择器

### 5. 启动页主题和工作台外观已经完成一次收敛

当前已经可以明确写成已实现的 UI 事实包括：

- 启动阶段会先按系统主题给出兜底外观
- 读取正式主题后再切换到配置中心主题
- 夜间模式已经进入冷启动阶段，而不是工作台加载完才切
- 聊天面板布局已经固定为消息区 + 底部输入区 + 工具栏

## 第二类：已经局部接通，但还不能夸大的部分

### 1. 设置页的大量表单仍然只是本地交互

这是当前最容易被误判的地方。

设置页现在确实已经有正式入口，但正式接入的只是少量关键字段。

仍主要是本地交互或页面组织的内容包括：

- 模型服务商编辑器
- 默认模型页中的主助手模型 / 快速执行模型路由
- 常规设置中的语言、代理、通知、自动备份
- 显示设置中的字号与界面密度
- 数据、MCP、搜索、记忆、文档处理页面的大多数项
- API 页面中的后端地址、重连策略、健康检查轮询

所以现在不能因为设置页“看起来很完整”，就把整页都写成已进入统一配置中心。

### 2. `model` 已经接入，但不能被夸大成“整套后端模型配置已经完成”

当前 `backendExposed.model` 的准确定位是：

- 已正式进入配置中心
- 已有设置页入口
- 已会被主进程读取
- 已会在下次完整启动时投影给后端 runtime

但它仍然不等于：

- 聊天面板每次消息最终使用的模型
- 完整的模型路由产品能力
- 整套后端可暴露字段方案已经做完

### 3. 会话列表已经可用，但还不是完整会话产品

当前中间会话列已经能：

- 创建会话
- 在当前窗口里保存会话列表
- 切换激活会话

但它还没有做到：

- 跨刷新恢复
- 从后端拉历史会话目录
- 自动回放历史消息

所以它现在已经不是占位，但也还不是完整会话管理产品。

## 第三类：当前仍主要是结构占位的部分

### 1. `capabilities`

当前主要作用是：

- 提供未来能力页的信息架构和布局容器

### 2. `files`

当前主要作用是：

- 提供未来文件工作区的信息架构和布局容器

### 3. `developer`

当前主要作用是：

- 提供未来开发工作台的信息架构和布局容器

这三页目前共同特点很明确：

- 有入口
- 有结构
- 但没有真实数据回流
- 也没有正式持久化闭环

## 可以把当前前端分成三层成熟度

### 第一层：已经进入正式主链路

- 统一配置中心主服务
- 主题 / 动画 / runtime override / 后端模型等正式字段
- 根装配层 bootstrap 状态装配
- 助手工作区的 session-first 聊天主路径
- 聊天面板当前布局和请求级模型 / 工具选择 UI

### 第二层：已经有界面和局部行为，但还没完全做实

- 设置页中的大多数分区
- 当前窗口内存态会话列表
- 聊天消息区的历史展示能力
- 默认模型页中的模型路由 UI

### 第三层：仍然主要是未来承载结构

- `capabilities`
- `files`
- `developer`
- 设置页里大量未进入正式配置链路的字段

## 接下来最清晰的方向

### 1. 继续补主路径，而不是继续扩表单面积

当前最值得继续补实的，是已经进入正式主路径的部分：

- 配置中心正式字段
- runtime 连接判断
- 智能体目录
- 会话创建与切换
- 请求级模型 / 工具执行

### 2. 继续按 owner 清楚的字段接配置中心

当前统一配置中心已经证明这条路线是可行的。

接下来更合理的是：

- 挑真正需要长期生效的字段继续接入
- 而不是把所有看起来完整的表单一次性都写进正式 schema

### 3. 继续把“默认模型字段”和“消息请求模型”分开写清楚

这是当前最容易混淆、也最需要文档持续保持一致的地方之一。

### 4. 继续把“会话壳可用”和“完整会话产品完成”分开写清楚

当前会话壳已经进入主路径，但还没做到完整恢复和回放能力。后续文档应继续保持这个边界，不要提前写满。

## 当前不要再写成这些说法

下面这些说法现在都不准确：

- “前端当前最成熟的仍然只是桌面壳和聊天骨架。”
- “统一配置中心还只是未来方向。”
- “聊天仍然主要依赖全局 agentName。”
- “模型选择器和工具选择器只是界面收敛，没有真实请求语义。”
- “设置页已经整体接完正式配置。”
- “助手工作区左侧和中间列仍然都是纯占位。”
