# 已实现、占位与下一步

这篇文档不讲细节实现，而是帮读者快速判断一件事：

> 当前前端到底哪些已经是稳定事实，哪些只是局部接通，哪些还只是页面结构。

如果你准备继续开发，先把这三类边界分清，会比直接盯着页面外观更有用。

## 先给结论

当前前端已经不再只是桌面壳或页面草图，而是有三条明确主线：

1. **统一配置中心已经正式落地**。
2. **聊天主路径已经切到 session-first**。
3. **工作台 UI 已经完成一轮现代化收敛**。

但这并不代表：

- 设置页所有字段都已经接好；
- 聊天已经具备完整历史回放和会话恢复；
- 其余工作区都已经有真实业务数据。

## 第一类：当前可以放心写成已实现的部分

### 1. Electron 桌面工作台已经稳定存在

当前已经明确存在：

- Electron 主进程
- preload 预加载层
- React renderer 工作台
- 左侧图标栏与多工作区结构
- 根级启动壳与错误兜底

这说明当前前端已经是可以持续演进的桌面应用，不再只是页面原型。

### 2. 统一配置中心已经是一条正式主链路

当前统一配置中心已经具备：

- 分域多文件 JSON
- 可读磁盘文件
- 主进程归一化
- legacy migration
- 公共快照 / 公共补丁 / 更新订阅

当前正式字段包括：

- `theme`
- `animationsEnabled`
- `agentName`
- `runtimeUrl`
- `model`

其中：

- `theme`、`animationsEnabled` 会立即影响前端显示
- `runtimeUrl` 是开发态 override 字段
- `model` 会影响下一次后端启动时的默认模型投影
- `agentName` 仍然持久化，但已不再是聊天 readiness 的硬门槛

### 3. 助手工作区已经进入 session-first 聊天主路径

当前聊天主路径已经可以写成：

1. 先确认有可用 runtime URL
2. 拉后端智能体目录
3. 选择智能体
4. 创建会话
5. 拉取会话能力面
6. 发送消息，并在请求里显式带上模型和工具选择

这意味着当前前端已经不再靠“全局 agent + 旧聊天桥”组织主入口。

### 4. 当前聊天 UI 已经有一套稳定外观

当前聊天面板已经具备：

- 状态分层清楚的启动 / 错误 / 就绪提示
- 三栏助手工作区布局
- 可滚动消息区
- 底部输入区
- 发送按钮
- 模型选择器
- 工具选择器

### 5. 启动页主题和工作台外观已经完成一次收敛

当前已经可以明确写成已实现的 UI 事实包括：

- 启动阶段会先按系统主题给出兜底外观
- 读取正式主题后再切换到配置中心主题
- 夜间模式已经进入冷启动阶段，而不是工作台加载完才切
- 聊天面板布局已经固定为消息区 + 底部输入区 + 工具栏

## 第二类：已经持久化，但尚未大范围接入真实业务的部分

### 1. 设置页已经不是“主要靠本地交互”的状态

这是当前最容易写错的地方。

设置页现在已经同时存在两条正式持久化链路：

- 统一配置中心负责公开字段，例如 `theme`、`animationsEnabled`、`runtimeUrl`、`model`、`agentName`
- settings workspace 负责设置页大部分工作区字段的保存与恢复，例如 SUSTech 信息、provider profiles、默认模型路由、常规设置、数据、MCP、搜索、记忆、API、文档处理、外部源等
- provider API key 与 CAS password 也会落盘，但它们进入的是 settings workspace 的 secrets 文档，由 Electron 主进程单独管理

因此，当前准确说法应该是：

- 设置页里已经有一批字段可以保存并恢复
- 其中只有一部分字段已经接入真实运行语义
- 另一大批字段虽然已经持久化，但目前主要还是为后续业务接线保留入口

把整个设置页继续写成“仍然只是本地交互”已经不符合现状。

### 2. 可以把设置页字段拆成三种状态

#### 已正式持久化，而且已经有明确生效路径

这类字段既能保存，也已经有比较清楚的当前作用：

- 公开配置中心中的 `theme` 与 `animationsEnabled` 会直接影响前端显示
- 公开配置中心中的 `runtimeUrl` 会作为开发态 runtime override 入口
- 公开配置中心中的 `model` 会在下一次完整启动时由主进程投影给后端 runtime 默认模型
- settings workspace 中的 provider profiles、默认模型路由、SUSTech 基本信息、常规设置、API 页面字段等都能在重新打开页面后恢复
- settings workspace 中的 provider API key 与 CAS password 也能恢复，但它们属于 secret owner 范围，不会进入公开快照

#### 已经持久化，但当前主要还是设置资产而不是现行真源

这一类字段最容易被误读成“已经全面生效”：

- 模型服务商编辑器会把 provider profile、模型目录、默认模型和快速模型路由写入 settings workspace
- 默认模型页中的主助手模型与快速执行模型现在会保存，但聊天面板发送消息时仍然要求显式给出本次 `model`
- 常规设置中的语言、代理、通知、自动备份目前主要体现为设置资产保留，尚未看到完整业务闭环
- 数据、MCP、搜索、记忆、文档处理页面中的大多数字段已经会保存，但还没有形成成体系的运行时接线
- API 页面中的重连策略、健康检查轮询、地址草稿会保存，但它们不等于 Python runtime 已直接采用这些值
- SUSTech 学号、邮箱、Blackboard 自动下载开关和下载上限会保存，但相关后端消费链路仍然有限
- 外部源里的 WakeUP 分享链接会保存，但“解析链接”仍然只是前端占位式流程

#### 仍然主要是占位或后续接线目标

这类内容当前仍然只能写成后续工作：

- WakeUP 链接解析还没有接入真实后端解析能力，当前对话框只是前端占位反馈
- 更多业务能力还没有把 settings workspace 中的大量字段真正接到运行时或业务服务
- 某些会话工具策略、模型偏好细化和跨页面联动，仍然需要继续补主路径

### 3. `model` 已经接入，但不能被夸大成“整套后端模型配置已经完成”

当前 `backendExposed.model` 的准确定位是：

- 已正式进入配置中心
- 已有设置页入口
- 已会被主进程读取
- 已会在下次完整启动时投影给后端 runtime

但它仍然不等于：

- 聊天面板每次消息最终使用的模型
- 完整的模型路由产品能力
- 整套后端可暴露字段方案已经做完

### 4. 会话列表已经可用，但还不是完整会话产品

当前中间会话列已经能：

- 创建会话
- 在当前窗口里保存会话列表
- 切换激活会话

但它还没有做到：

- 跨刷新恢复
- 从后端拉历史会话目录
- 自动回放历史消息

所以它现在已经不是占位，但也还不是完整会话管理产品。

## 第三类：当前仍主要是结构占位的部分

### 1. `capabilities`

当前主要作用是：

- 提供未来能力页的信息架构和布局容器

### 2. `files`

当前主要作用是：

- 提供未来文件工作区的信息架构和布局容器

### 3. `developer`

当前主要作用是：

- 提供未来开发工作台的信息架构和布局容器

这三页目前共同特点很明确：

- 有入口
- 有结构
- 但没有真实数据回流
- 也没有形成像设置页那样清楚的正式持久化闭环

## 可以把当前前端分成三层成熟度

### 第一层：已经进入正式主链路

- 统一配置中心主服务
- 主题 / 动画 / runtime override / 后端模型等正式字段
- settings workspace 的状态文档与 secrets 文档
- 根装配层 bootstrap 状态装配
- 助手工作区的 session-first 聊天主路径
- 聊天面板当前布局和请求级模型 / 工具选择 UI

### 第二层：已经能保存和恢复，但还没大范围接入业务真源

- 设置页中的大多数分区
- provider profiles 与默认模型路由
- SUSTech 信息、API、数据、MCP、搜索、记忆、文档处理、外部源等 settings workspace 字段
- 当前窗口内存态会话列表
- 聊天消息区的历史展示能力
- 默认模型页中的模型路由 UI

### 第三层：仍然主要是未来承载结构

- `capabilities`
- `files`
- `developer`
- WakeUP 解析后端与更多业务接线

## 接下来最清晰的方向

### 1. 继续补主路径，而不是继续扩表单面积

当前最值得继续补实的，是已经进入正式主路径的部分：

- 配置中心正式字段
- runtime 连接判断
- 智能体目录
- 会话创建与切换
- 请求级模型 / 工具执行

### 2. 继续按 owner 清楚的字段接配置链路

当前统一配置中心和 settings workspace 已经证明这条路线是可行的。

接下来更合理的是：

- 挑真正需要长期生效的字段继续接入合适的 owner
- 继续区分公开快照字段、settings workspace 状态字段与 secret 字段
- 而不是因为页面已经长出来，就把所有字段都写成“已经全面生效”

### 3. 继续把“默认模型字段”和“消息请求模型”分开写清楚

这是当前最容易混淆、也最需要文档持续保持一致的地方之一。

### 4. 继续把“会话壳可用”和“完整会话产品完成”分开写清楚

当前会话壳已经进入主路径，但还没做到完整恢复和回放能力。后续文档应继续保持这个边界，不要提前写满。

## 当前不要再写成这些说法

下面这些说法现在都不准确：

- “前端当前最成熟的仍然只是桌面壳和聊天骨架。”
- “统一配置中心还只是未来方向。”
- “聊天仍然主要依赖全局 agentName。”
- “模型选择器和工具选择器只是界面收敛，没有真实请求语义。”
- “设置页的大多数内容仍然只是本地交互，没有正式持久化。”
- “设置页已经整体接完正式配置。”
- “settings workspace 里保存的字段都已经直接进入聊天请求或 Python runtime 真源。”
- “助手工作区左侧和中间列仍然都是纯占位。”
