# 已实现、占位与下一步

## 这篇文档适合谁看

这篇文档适合：

- 想快速判断“现在到底做了多少”的前端开发者
- 需要区分“当前事实”和“前端占位”的联调参与者
- 想判断统一配置中心当前已经收编到哪一步的人

如果你想先看具体字段和状态，请直接看：

- [当前生效字段参考](./reference-current-fields.md)
- [前端运行时状态参考](./reference-runtime-states.md)
- [页面能力参考](./reference-page-capabilities.md)

## 先给结论

- 当前前端最成熟的部分，已经不只是 Electron 桌面壳和聊天入口骨架，还包括一条正式落地的统一配置中心主链路。
- 当前已经正式纳入统一配置中心的字段共有 4 个：`theme`、`agentName`、`runtimeUrl`、`model`。
- 但这 4 个字段的成熟度并不一样：`theme`、`agentName`、`runtimeUrl` 已有更完整的 UI / 行为闭环，`model` 目前仍是主进程投影样板字段。
- 设置页虽然已经开始承载正式配置入口，但大多数字段仍然只是前端本地交互，不应当算作已生效配置。
- 下一步更合理的主线，仍然是继续补主路径和关键字段，而不是把未稳定的整页表单一次性都写进正式 schema。

## 当前已经实现的部分

### 1. 桌面应用壳已经稳定存在

当前前端已经具备完整的 Electron 桌面壳：

- 主进程
- preload 预加载层
- renderer 工作台
- 多工作区入口
- 基本构建和打包链路

这说明项目已经不是纯页面草图，而是有持续演进空间的桌面宿主环境。

### 2. 工作台结构已经比较清晰

当前已经形成明确的工作台结构：

- `assistant`
- `capabilities`
- `files`
- `developer`
- `settings`

其中：

- `assistant` 已有三段式布局
- `settings` 已有独立布局和较完整交互
- 另外三个工作区已形成独立页面结构

### 3. 统一配置中心主链路已经真正落地

这是当前和旧文档相比最需要更新的地方。

现在已经存在这样一条正式链路：

1. 主进程按域读取统一配置中心文档
2. preload 暴露公共快照、公共补丁和公共快照订阅接口
3. renderer 通过公共快照读取 bootstrap fields
4. 设置页中的少量正式字段通过公共补丁写回配置中心
5. 主进程写盘后广播公共快照更新
6. renderer 根据更新重新同步主题或重新计算 bootstrap 状态

这意味着当前前端已经不再只是“未来要做配置中心”，而是已经有一个真实可用的最小切片。

### 4. 当前已经正式接入的字段

当前正式字段有：

- `theme`
- `agentName`
- `runtimeUrl`
- `model`

但要分开理解：

#### 已有 UI / 行为闭环的字段

- `theme`
- `agentName`
- `runtimeUrl`

#### 已进 schema 和主进程投影链路，但当前无正式 UI 的字段

- `model`

因此，当前可以说“统一配置中心已经覆盖四个稳定域的最小样板字段”，但不能说“后端可暴露字段样板已经整体完成”。

### 5. 聊天入口已经有真实状态主线

当前聊天区最重要的价值，不只是“有个聊天区域”，而是它已经能根据实际连接状态给出不同表现。

当前已经稳定存在的状态主线包括：

- `loading`
- `empty`
- `incomplete`
- `starting`
- `ready`
- `failed`
- `degraded`
- `error`

而这条主线现在已经和统一配置中心正式连上：

- `agentName` 与 `runtimeUrl` 来自配置中心公共快照
- hosted runtime 事实来自主进程 runtime 快照
- 最终由 renderer 统一归并

## 当前只是前端占位或部分接入的部分

### 1. 设置页的大多数分区还没有正式收编完成

这是现在最容易被误解的地方。

虽然设置页已经开始承载正式配置入口，但当前正式接入的只是少量关键字段。

大量仍主要停留在前端本地 state 的内容包括：

- 模型服务
- 默认模型 UI
- 常规设置中的语言、代理、通知等字段
- 显示设置中的字号、密度、动画
- 数据设置
- MCP
- 搜索
- 记忆
- API 服务器页里的多数占位字段
- 文档处理

所以现在不能因为设置页看起来很完整，就把整页都当成“已纳入统一配置中心”。

### 2. `model` 不能被夸大成“已经完成的后端可暴露字段方案”

当前 `model` 的准确定位是：

- 已经进入统一配置中心 schema
- 已经进入主进程读取和 runtime 参数投影链路
- 但当前没有正式 UI 入口
- 也没有把“后端可暴露字段”这整类能力做成完整产品能力

因此，当前最准确的表述应该是：

- **后端可暴露域已有最小样板字段接入**

而不是：

- **后端可暴露字段样板已经全部完成**

### 3. API 服务器页现在是“部分正式接入 + 部分占位”

这一页当前最容易让人误判。

它已经正式接入的内容有：

- `runtimeUrl` 卡片
- 根层 bootstrap 状态摘要
- 根层统一 retry 动作

但它还没有正式接通的内容有：

- 后端地址占位字段
- 重连策略
- 健康检查轮询

所以这页当前已经不是纯占位，但也远不是完整 API 设置产品页。

### 4. 三个扩展工作区仍然主要是结构占位

`capabilities`、`files`、`developer` 三个工作区当前的准确定位仍然是：

- 有结构
- 有主题
- 有页面组织
- 但没有真实数据回流

所以它们主要还是为未来真实页面提供承载结构。

## 可以把现状分成三类

### 第一类：当前代码事实，可以放心写成已实现

- Electron 桌面壳
- 多工作区结构
- 根装配层统一 bootstrap 状态装配
- 统一配置中心主服务
- 公共快照 / 公共补丁 / 公共订阅桥接
- `theme`、`agentName`、`runtimeUrl` 的设置页正式入口
- `model` 进入 schema 和主进程 runtime 投影链路
- 主题即时生效
- 聊天入口按状态切换 UI

### 第二类：已经有界面或局部接入，但还不能夸大

- 设置页里的大多数字段
- 默认模型页的模型路由 UI
- API 服务器页里的多数表单项
- `model` 的完整产品化入口
- `assistant` 左侧助手与话题区域

### 第三类：仍属于未来方向

- 设置页大面积字段的正式收编
- 更完整的 runtime 状态联动
- 完整聊天产品化
- 真实文件数据工作区
- 真实开发工作区数据
- 更成体系的后端可暴露字段集合

## 下一步最清晰的实现方向

### 1. 继续补主路径，而不是先扩占位面

当前最有价值的主路径仍然是：

- 配置中心正式字段
- bootstrap 状态判断
- hosted runtime 连接
- 最小聊天入口

所以接下来更合理的方向，仍然是把这条链继续补实。

### 2. 继续按“稳定 owner + 最小切片”收编设置字段

当前统一配置中心已经证明：

- 分域多文件可行
- 主进程聚合可行
- 公共快照 / 补丁可行

下一步更合理的是：

- 继续挑真正需要长期生效的字段接入
- 而不是把整页还没稳定的表单一次性都写进正式 schema

### 3. 把“已经有样板”和“已经做完”分清楚

这点对后续文档尤其重要。

例如：

- `model` 现在已经是样板字段
- 但后端可暴露字段方案还没有完成

后续推进时应继续沿用这种写法，避免把设计阶段设想提前写成已实现事实。

## 当前边界 / 不要误解的地方

- 当前前端已经不只是页面原型，而是有真实配置主链路的桌面应用。
- 当前设置页已经开始承载正式配置入口，但远没有整页都收编完成。
- 当前统一配置中心已经做到四个稳定域的最小字段切片，但这不等于所有设置都已稳定入表。
- 当前旧 `copilot-settings.json` 只保留主进程内部迁移语义，不应再被写成 renderer 正式接口。
