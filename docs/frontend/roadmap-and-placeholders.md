# 路线图与占位边界

## 文档目的

本文档用于系统说明当前前端中“已经落地的能力”和“仍处于占位或计划中的能力”之间的边界，避免在评审、联调或后续开发时把演示性界面误认为已完成实现。

本文聚焦以下问题：

- 当前前端到底已经实现了什么
- 哪些部分只是骨架、占位或前端演示交互
- 下一阶段最明确的补齐方向是什么
- 在讨论开发计划时，哪些内容可以作为当前基础继续推进

## 一、当前已经实现的能力

基于当前代码，可以明确归入“已实现”的部分如下。

### 1. 桌面应用壳层

前端已经具备 Electron 桌面壳层，可作为桌面端应用启动、开发与打包的基础载体。Electron 相关代码位于 `frontend-copilot/electron/`，打包配置位于 `frontend-copilot/electron-builder.json5`。

这意味着：

- 当前项目不是纯浏览器页面原型
- 已具备主进程、preload 与 renderer 的基本分层
- 后续可在现有桌面壳层中继续接入系统级能力与本地持久化能力

### 2. Renderer 应用骨架

Renderer 入口链路已经明确：

- `frontend-copilot/src/main.tsx`
- `frontend-copilot/src/CopilotAppRoot.tsx`
- `frontend-copilot/src/App.tsx`

这条链路已经形成稳定的启动路径，可作为后续继续扩展 UI 与 Copilot 集成的主入口。

### 3. Copilot Provider 条件包裹逻辑

当前前端已经实现基于配置状态的 Copilot Provider 条件包裹逻辑：只有当配置状态为 `ready` 时，才会在应用外层包裹 CopilotKit。

这部分属于当前最关键的“已实现事实”之一，因为它说明：

- Copilot 运行时接入不是无条件注入
- 应用已经区分“未配置”“配置不完整”“配置就绪”“读取异常”
- 前端启动链路中已经存在正式的配置判定步骤

### 4. Copilot 设置持久化底层链路

当前 Electron 与 renderer 之间已经具备 Copilot 设置的读取 / 保存底层链路，并且已经明确：

- 配置文件名为 `copilot-settings.json`
- 存储位置在 Electron `userData` 目录下
- preload 向 renderer 暴露加载与保存接口
- renderer 侧已有统一调用封装

需要强调：这并不等于“设置页所有配置项都已持久化”，而是指 Copilot 最小配置链路已经打通。

### 5. 工作区级别的桌面 UI 架构

当前界面已经形成稳定的工作区结构：

- assistant
- capabilities
- files
- developer
- settings

其中：

- `assistant` 工作区已采用三段布局
- `settings` 工作区已采用独立布局
- 其余工作区已有独立入口与结构化占位内容

这说明前端的信息架构已经先于真实业务能力完成初步落地。

### 6. 设置页基础交互

当前设置页中已经存在大量可点击的前端交互，包括：

- 开关切换
- 下拉展开与选择
- 文本输入
- 多分区导航
- 服务商列表切换与新增

这些交互已经足以支撑界面评审、信息架构讨论与后续逐项接入真实配置逻辑。

## 二、当前仍属于占位或演示的部分

以下内容虽然已经出现在界面或代码结构中，但当前不能写成“已经完成”。

### 1. 真实聊天 UI

当前尚未接入真实的 `@copilotkit/react-ui` 聊天界面。现有聊天区域仍是配置状态说明与占位面板。

这意味着当前尚未完成：

- 正式消息列表渲染
- 对话输入与发送
- 流式返回展示
- 历史会话管理
- 完整聊天体验

因此，不能把当前界面描述为“已上线 Copilot 聊天功能”，更准确的表述应是“聊天区域骨架已就绪”。

### 2. 设置页与 Electron 持久化的全面接通

虽然设置页中存在大量字段，但当前真正接入 Electron 持久化并参与 Copilot 启动判断的字段只有：

- `runtimeUrl`
- `agentName`

除这两个字段之外，设置页中的“API 服务器”“模型服务商”“默认模型”“网络搜索”“全局记忆”等大多仍是前端本地 state 演示。

因此，这些设置项当前应被视为：

- 已有界面外观与交互骨架
- 尚未具备统一持久化能力
- 尚未形成已生效业务配置

### 3. 后端健康检查

当前尚未接入后端健康检查链路。

因此文档中不能写成：

- 已实现自动探测后端在线状态
- 已支持后端自动重连策略生效
- 已形成前端可验证的服务可用性判断

设置页中与 API 服务器相关的展示，更适合被理解为后续计划中的界面占位。

### 4. capabilities / files / developer 的真实数据功能

这三个工作区当前都已有标题、说明、亮点与条目结构，但仍主要用于表达未来方向，尚未接入真实数据。

具体来说：

- `capabilities`：尚未展示真实工具注册、在线状态或能力调用结果
- `files`：尚未展示真实文件索引、上传附件、知识库数据或检索结果
- `developer`：尚未展示真实日志、调试任务、执行记录或开发面板数据

因此，这三个工作区目前都应归类为“结构已搭建、功能仍占位”。

## 三、当前最清晰的下一阶段实现方向

在不超出当前代码事实的前提下，可以将下一阶段的前端工作重点概括为以下几个方向。

### 1. 把聊天骨架升级为真实聊天界面

这是当前最直接的一步，包括：

- 接入真实聊天 UI 组件
- 让 `ready` 状态不止显示说明面板，而是承载真正的对话体验
- 建立聊天内容与运行时响应之间的前端展示闭环

这是把当前“Copilot 已接入基础框架”推进到“用户可实际对话”的关键步骤。

### 2. 把设置页占位字段逐步接入持久化

下一阶段应把现有设置页中的关键字段按优先级接入 Electron 持久化，而不是只停留在本地 React state。

从当前代码边界来看，这项工作至少包括：

- 明确哪些设置需要真正保存
- 为这些设置建立统一 bridge 与存储结构
- 让界面中的编辑结果能在重启后保留

### 3. 补齐后端可用性校验

如果后续要支持稳定联调，前端还需要补齐与后端运行时相关的最基础校验能力，例如：

- 配置完成后的连通性反馈
- 更明确的读取失败 / 配置错误 / 服务不可达区分
- 与“未配置”状态不同的运行时异常提示

当前代码已经区分了配置读取异常与未配置状态，这为后续继续细化运行时错误反馈打下了基础。

### 4. 将占位工作区接入真实数据源

当前 `capabilities`、`files`、`developer` 都已经有较清楚的信息架构。后续可在这些结构上逐步接入真实数据，而不必从零设计界面。

可预期的发展方向包括：

- 能力中心接入真实工具列表与状态
- 文件工作区接入资料、附件与知识索引
- 开发工作区接入执行记录、调试信息与可观测性内容

这里应理解为“前端结构已准备好，等待真实能力填充”。

### 5. 明确区分“真实生效配置”与“展示性配置”

随着设置项逐步增多，后续开发中必须始终保持一个原则：

- 文档与实现都要明确说明哪些字段已经生效
- 未接通的展示性字段必须继续标注为占位
- 前后端联调只应围绕当前真实生效的契约展开

这对于避免误解、降低联调成本非常重要。

## 四、推荐的优先级顺序

结合当前代码现状，后续实现顺序可按下列逻辑理解：

1. 优先保证 Copilot 最小连接闭环稳定
2. 再把聊天区域从状态面板升级为真实聊天 UI
3. 再逐步把关键设置项接入持久化与校验
4. 最后将能力、文件、开发等工作区接入真实业务数据

这种顺序的优点是：

- 与现有代码结构一致
- 能最早形成可联调、可演示、可验证的主路径
- 不会过早在占位模块上投入大量实现成本

## 五、文档使用边界

在阅读或引用本文档时，应特别注意：

- “已实现”表示当前代码中已经存在可确认的实际能力
- “占位”表示界面或结构已出现，但真实业务功能尚未接入
- “未来计划”表示依据当前结构与边界可明确推导出的下一阶段方向

因此，本文中的“计划”不是已经完成的事实，也不等同于后端契约。涉及当前可联调边界时，应优先参考 `backend-connection-contract.md`。

## 六、小结

当前前端最准确的阶段判断是：

- 基础桌面壳层已经完成
- UI 信息架构已经基本建立
- Copilot 最小配置链路已经打通
- 但真实聊天、设置全面持久化、后端健康检查与多个工作区的数据能力仍处于占位或下一阶段实现范围内

因此，本项目当前更适合被描述为“可持续迭代的前端框架与工作台骨架”，而不是“功能已完整落地的桌面 Copilot 产品”。
