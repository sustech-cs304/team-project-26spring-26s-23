# 已实现、占位与下一步

## 这篇文档适合谁看

这篇文档适合：

- 想快速判断“现在到底做了多少”的前端开发者
- 需要区分“当前事实”和“前端占位”的联调参与者
- 准备规划下一步前端实现顺序的人

如果你还没看过界面说明，建议先读 [ui-current-state.md](./ui-current-state.md)。如果你想查字段或运行态，请直接看附录：

- [reference-current-fields.md](./reference-current-fields.md)
- [reference-runtime-states.md](./reference-runtime-states.md)
- [reference-page-capabilities.md](./reference-page-capabilities.md)

## 这篇文档回答什么问题

这篇文档主要回答：

- 当前前端已经真正实现了什么
- 哪些内容只是前端界面骨架、交互演示或占位页面
- 下一步最值得优先补齐的方向是什么
- 在讨论路线时，哪些内容能当当前基础，哪些只能当未来规划

## 先给结论

- 当前前端最成熟的部分，是 Electron 桌面壳、React 工作区结构、Copilot 最小配置读取链路，以及按配置状态切换的聊天面板骨架。
- 当前真正能作为“已生效事实”依赖的 Copilot 配置，只有 `runtimeUrl` 和 `agentName` 两个字段。
- 设置页虽然已经很像完整产品，但里面大多数字段目前仍然只是前端本地交互，不应当作已生效配置。
- `capabilities`、`files`、`developer` 三个工作区现在主要用于表达未来信息架构，仍属于占位工作区。
- 下一步最清晰的主线，不是继续堆更多占位页面，而是先把聊天主路径、关键配置持久化和更明确的运行时反馈补起来。

## 再展开说明

## 当前已经实现的部分

### 1. 桌面应用壳已经存在

当前前端不是普通网页原型，而是一个 Electron 桌面应用。它已经具备：

- 主进程
- preload 预加载层
- renderer 界面层
- 基本的构建和打包链路

这意味着当前项目已经有了可持续扩展的桌面宿主环境，后面要继续接本地存储、系统能力或桌面级交互，都有基础可接。

### 2. 工作台结构已经比较稳定

当前前端已经形成一个比较明确的桌面工作台结构，主要工作区包括：

- `assistant`
- `capabilities`
- `files`
- `developer`
- `settings`

其中：

- `assistant` 已经有三段式布局
- `settings` 已经有独立布局
- 另外三个工作区已经有独立入口和各自的页面结构

这部分的意义在于：虽然功能还不完整，但产品信息架构已经不再是空白。

### 3. Copilot 最小读取链路已经打通

当前最重要的“已实现事实”之一，是前端已经能在启动时读取本地 Copilot 配置，并根据结果判断当前状态。

目前这条最小链路包括：

- Electron 从本地文件读取配置
- preload 向 renderer 暴露加载和保存接口
- renderer 侧统一封装读取结果
- 应用启动时根据配置状态决定是否初始化 Copilot 外层能力

这说明项目已经不只是“画了一个聊天框”，而是已经有真正参与启动逻辑的配置判断。

### 4. 聊天区域已经有运行态骨架

当前聊天区域虽然还不是完整聊天 UI，但已经能区分不同运行态，并给出不同提示。

现在已经存在的运行态包括：

- `loading`
- `empty`
- `incomplete`
- `ready`
- `error`

这部分很重要，因为它已经把“还没配”“只配了一半”“读取失败”“最小条件齐了”这些情况分开了。后面继续做真实聊天体验时，可以直接沿用这条状态主线，而不用重新设计整个入口逻辑。

### 5. 设置页的前端交互已经比较完整

当前设置页已经有相当多的前端交互：

- 分区导航
- 输入框
- 下拉框
- 开关
- 服务商列表切换
- 新增服务商

这些能力的价值在于：后续接真实配置时，很多页面组织和交互形式已经有现成承载体，不需要从零开始画设置中心。

## 当前只是前端占位或前端交互的部分

### 1. 真实聊天功能还没有完成

现在最容易被误解的地方，就是右侧聊天区。

当前它更准确的定位是：**按配置状态切换的聊天面板骨架**。

这意味着当前还没有：

- 正式消息列表
- 输入框发起真实对话
- 流式响应展示
- 历史会话持久化
- 运行时交互后的完整聊天反馈

所以，当前可以说“聊天骨架已经有了”，但不能说“聊天功能已经完成”。

### 2. 设置页的大多数字段还没有正式生效

设置页里现在有很多看起来很完整的字段，例如：

- 模型服务
- 默认模型
- 常规设置
- 显示设置
- 数据设置
- MCP 服务器
- 网络搜索
- 全局记忆
- API 服务器
- 文档处理

但根据当前实现，大多数内容仍然只是前端本地 state。

白话说就是：

- 能点
- 能切
- 能改
- 但大多不会构成当前真正生效的配置能力

目前真正进入 Copilot 启动判断并明确生效的字段，仍然只有：

- `runtimeUrl`
- `agentName`

所以，设置页现在的准确定位应是：

- **当前事实**：页面结构和交互外观已实现
- **前端占位**：绝大多数字段还未形成正式持久化或后端联调能力

### 3. API 服务器相关内容仍然是占位说明

设置页里虽然有“API 服务器”分区，也有“后端地址”“重连策略”“健康检查轮询”等字段，但当前不能把它们写成已接通能力。

当前更准确的理解是：

- 页面上已经表达了未来可能需要的配置方向
- 但现在还没有真实健康检查闭环
- 也没有可依赖的自动重连实现说明
- 更没有可以确认的 HTTP 契约细节

因此，这一块目前仍属于“前端界面已摆出来，但还没有形成真实运行能力”。

### 4. 三个扩展工作区仍然主要是结构占位

当前 `capabilities`、`files`、`developer` 这三个工作区都有独立入口，也都有各自的标题、描述、亮点和条目。

但它们现在主要用于表达未来方向，而不是承载真实数据：

- `capabilities` 还没有真实工具状态或能力调用反馈
- `files` 还没有真实文件索引、上传结果或知识数据
- `developer` 还没有真实日志、任务状态或调试结果

所以这三页当前更像“以后要接什么”的结构说明，而不是“现在已经能用”的业务页面。

## 可以把现状分成三类

### 第一类：当前代码事实

这些内容可以放心写成“已经存在”：

- Electron 桌面壳
- 多工作区页面结构
- `assistant` 三段布局
- `settings` 独立布局
- Copilot 配置读取链路
- 按配置状态切换的聊天面板骨架
- `runtimeUrl` 与 `agentName` 这两个最小生效字段

### 第二类：当前只有前端交互，但未持久化或未接后端

这些内容现在不能夸大成已生效能力：

- 设置页中的绝大多数字段
- 服务商管理外观
- 默认模型路由外观
- 搜索、记忆、MCP、数据、显示等配置页
- API 服务器分区中的大多数展示项

### 第三类：未来方向或设计讨论内容

这些内容可以讨论，但不能写成当前事实：

- 真实聊天 UI
- 健康检查与重连闭环
- 能力中心接真实工具数据
- 文件工作区接真实文件和知识索引
- 开发工作区接真实日志和任务状态
- 更完整的后端接口组织方式

## 下一步最清晰的实现方向

### 1. 先补主路径：让聊天区从骨架变成真正能用

当前前端最有价值的主路径，是从配置读取 → 状态判断 → 进入聊天区。

所以下一步最值得优先补的，是：

- 让 `ready` 状态下不再只是展示说明面板
- 接入真实聊天 UI
- 让用户能够实际发起对话并看到返回结果

### 2. 再补关键配置：把真正需要长期生效的字段接入持久化

当前设置页里字段很多，但真正生效的很少。下一步更合理的做法，不是继续加更多展示字段，而是挑关键字段逐步接到底层存储和启动链路里。

这一步的目标应该是：

- 明确哪些字段真的需要保存
- 明确哪些字段真的参与启动或运行判断
- 让界面上改过的关键配置在重启后还能保留

### 3. 再补运行反馈：把“没配好”和“运行出错”分得更清楚

当前已经区分了未配置和读取失败，这是一个好起点。下一步可以继续细化：

- 配置齐了但服务不可达
- 运行时连接异常
- 健康检查失败
- 后端返回异常

但这些都应建立在当前真实代码实现之上，不能先在文档里当作已实现写出来。

### 4. 最后再把占位工作区逐步接成真实页面

当前三个扩展工作区的信息架构已经有雏形，所以后续应该是在既有结构上逐步接真实数据，而不是推倒重来。

这类工作更适合在主路径跑通后分阶段补齐。

## 当前边界 / 不要误解的地方

- 当前前端已经是“能继续开发的骨架”，不是“已经完成的大产品”。
- 现在很多页面看起来完整，只能说明前端外观和交互做出来了，不能说明真实业务能力已经接通。
- 文档里提到的“下一步”只是基于当前结构给出的实现方向，不等于已经排定或已经完成。
- 涉及当前后端连接边界时，应优先以 [backend-connection-contract.md](./backend-connection-contract.md) 和 [reference-current-fields.md](./reference-current-fields.md) 为准。
