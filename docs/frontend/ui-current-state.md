# 前端当前 UI 状态

这页只讲用户今天在桌面端真正会看到什么，以及这些界面各自承担什么角色。字段存储位置见 [当前生效字段参考](./reference-current-fields.md)，页面成熟度对照见 [页面能力参考](./reference-page-capabilities.md)。

## 当前工作台的总体结构

当前前端是一个运行在 Electron 宿主中的 React renderer。进入主工作台后，整体仍然是“左侧入口栏 + 右侧主内容区”的布局。

左侧入口栏当前包含五个工作区：

- 助手。
- 能力。
- 文件。
- 开发。
- 设置。

其中真正进入当前主路径的工作区，主要还是“助手”和“设置”。其余三个工作区已经有稳定的页面骨架，但当前更像结构化入口，而不是已经接上真实后端数据的业务页。

## 启动阶段用户会看到什么

应用启动后，前端会先进入根装配阶段。这个阶段里，界面会根据公开配置中心和 hosted runtime 快照判断当前状态，再决定是继续进入工作台，还是先停留在说明性状态页。

用户当前能看到的典型状态包括：

- 正在准备桌面界面。
- 正在连接本地服务。
- 宿主启动失败或读取链路失败后的可解释错误卡片。

这部分界面的重点，是把“前端还在装配”“宿主正在启动”“当前没有可用运行时地址”这些情况区分开，而不是直接把所有失败都写成同一种连接错误。

## 助手工作区当前长什么样

助手工作区已经是当前最清晰的一条真实主路径。页面主体采用三栏结构：

1. 左侧是智能体目录。
2. 中间是会话列表。
3. 右侧是聊天主内容区。

### 左侧智能体目录

左侧目录展示的是后端返回的智能体列表。前端当前负责目录展示、当前选中状态和基础交互；目录内容本身以后端返回为准。

### 中间会话列表

中间列当前已经支持这些动作：

- 为当前智能体创建会话。
- 在当前窗口内切换会话。
- 拖动排序会话。
- 打开会话右键菜单。

这说明“先选智能体，再建会话”已经是当前聊天主路径的一部分。会话列已经不是占位壳，但它目前仍然首先服务当前窗口里的会话操作。

### 右侧聊天主内容区

右侧区域会根据连接状态、目录状态和会话状态切换不同界面。进入可聊天分支后，这里已经具备完整的消息发送壳：

- 可滚动的消息区。
- 空会话提示。
- 底部输入区。
- 模型选择器。
- 工具选择器。
- 发送按钮。

### 当前模型选择器展示的是真实模型路由目录

当前模型选择器有一个很重要的前提：它展示的模型目录来自 settings workspace 里的 provider profiles，而不是静态内置默认模型。如果 settings workspace 里还没有任何 provider 或模型，这里会显示“尚未配置模型”的空状态，而不是自动补出示例模型。

用户在这里选中的，也不再是一个单独的模型字符串，而是一条完整模型路由。当前页面侧已经围绕下面这组信息工作：

- `providerProfileId`
- `provider`
- `endpointType`
- `baseUrl`
- `modelId`

这也是当前 UI 的一个关键事实：产品里已经不存在“单一 active provider 决定全部模型集合”的主线语义。聊天区看到的是跨 provider 的模型目录。

### 当前工具选择器仍然存在，但它不是本期主线重点

工具选择器展示的是当前会话能力面返回的工具目录、推荐工具和默认信息。发送时，前端会把用户当前选择的工具显式带进请求。

同时也要注意：当前流式主线的重点仍然是模型路由、宿主取密钥与文本流。工具事件在协议里保留了 `tool_event_reserved` 预留位，但真实工具生命周期并不是当前 UI 主线的一部分。

### 当前聊天区已经按流式 run 过程更新

当前聊天区不再等待一份整包响应后再一次性落屏。页面已经按 run 生命周期工作：

- 收到 `run_started` 时，会创建 assistant 占位项。
- 收到 `text_delta` 时，会持续增量更新 assistant 文本。
- 收到 `run_completed` 时，会定稿最终 assistant 文本，并记录最终解析出的模型路由。
- 收到 `run_failed` 或 `run_cancelled` 时，会进入对应终态。
- 收到 `run_diagnostic` 时，会保留非敏感诊断信息。

因此，用户现在看到的聊天响应，已经是实时流式推进的过程，而不是旧的“一次请求得到完整 assistant 文本”界面。

### 当前取消与失败在页面上的表现

当前页面会保留失败项或取消态提示，用于当前窗口里的可见反馈。与此同时，正式会话归档只在成功完成时发生。页面上的失败项和取消态草稿，不等于后端已经把 assistant 成功消息写入正式历史。

## 设置工作区当前长什么样

设置工作区已经不是几张零散卡片，而是一组固定导航下的完整工作区。左侧导航当前包含 12 个分区：

1. `SUSTech 信息`
2. `模型服务`
3. `默认模型`
4. `常规设置`
5. `显示设置`
6. `数据设置`
7. `MCP 服务器`
8. `网络搜索`
9. `全局记忆`
10. `API 服务器`
11. `文档处理`
12. `外部源`

这组导航本身已经说明两件事：

- 设置页覆盖范围已经明显扩大。
- 页面结构已经稳定到可以承载长期设置，而不是一次性演示界面。

### 当前最成熟的几个设置分区

`模型服务` 是当前最完整的设置页之一。用户现在可以看到服务商列表、搜索、添加、复制、删除、拖动排序、详情编辑、API 密钥输入，以及模型列表和模型编辑弹层。

`默认模型` 当前聚焦两条默认路由，也就是主助手模型和快速执行模型。它和聊天面板的关系已经比较直接，因为聊天发送区会从 settings workspace 里读取默认模型路由。

`显示设置` 当前主要提供主题入口。界面角色很明确，就是控制桌面端外观主题。

`API 服务器` 当前同时承载两类内容：一类是公开配置中心里的开发态运行时覆盖地址，另一类是 settings workspace 里的地址草稿、重连策略和轮询字段。因此它看起来更像一个混合页。

### 当前已经可见但仍有明显边界的分区

`SUSTech 信息` 已经是正式页面，页面里可以看到校园账号信息、CAS 密码输入、Blackboard 相关偏好，以及一块 `TIS 信息` 区域。这个页面已经存在，但其中并不是每一块都已经接成成熟业务链路。

`外部源` 也已经有真实入口。用户当前可以看到 WakeUP 分享链接输入框、解析按钮和解析结果弹窗。不过这条流程目前仍然主要停留在前端侧的半接通状态，不能把它写成已经完成真实后端接入的成熟能力。

`常规设置`、`数据设置`、`MCP 服务器`、`网络搜索`、`全局记忆` 和 `文档处理` 都已经有稳定页面与表单结构，但它们更适合作为长期设置工作区来理解，而不是全部都已经变成前端主路径里的强生效项。

## 其余三个工作区当前是什么状态

`能力`、`文件` 和 `开发` 这三个工作区现在都已经有统一的 Hub 页面布局。页面里有标题、分组、卡片和高亮区域，因此信息架构已经稳定。

不过它们当前主要承载的是主题入口和结构位置，还没有像助手工作区那样接上真实目录、会话或运行时数据。因此，文档更适合把它们写成“已有结构的工作区页面”，而不是已经成熟的数据面板。

## 当前 UI 里容易被误读的几件事

- 设置页里有很多字段能够保存，并不意味着这些字段都已经接到真实运行时。
- 聊天区现在有模型选择器，并不意味着前端内置了一套默认 provider 或默认模型。首次状态可以是空白模型目录。
- 聊天区现在是流式渲染主线，不再只是消费非流式完整响应。
- `外部源` 页面已经存在，并不意味着 WakeUP 已经打通稳定后端流程。
- `能力`、`文件` 和 `开发` 已经有页面，并不意味着它们已经具备成熟的数据回流。
- 仓库里保留的 `copilot` 命名主要是历史沿用的内部命名空间，并不意味着当前 UI 主线仍然建立在第三方 CopilotKit runtime 上。
