# 当前界面现状导览

## 这篇文档适合谁看

这篇文档适合：

- 刚接手前端，想先看懂界面结构的人
- 需要区分“哪些区域已经能交互、哪些只是展示骨架”的人
- 准备继续接聊天、设置或工作区数据的人

如果你更关心字段、状态或工作区能力的查表信息，可以直接跳到附录：

- [reference-current-fields.md](./reference-current-fields.md)
- [reference-runtime-states.md](./reference-runtime-states.md)
- [reference-page-capabilities.md](./reference-page-capabilities.md)

## 这篇文档回答什么问题

这篇文档主要回答下面这些问题：

- 当前桌面前端界面由哪几块组成
- `assistant`、`settings`、`capabilities`、`files`、`developer` 五个工作区现在分别是什么状态
- 聊天区域现在是真聊天，还是只是状态面板
- 设置页里哪些是可交互外观，哪些已经真的接到底层配置链路

## 先给结论

- 当前前端已经有比较完整的“工作台式”界面结构，也就是一个带左侧导航、多个工作区的桌面工作台，而不是单页聊天原型。
- `assistant` 是现在最完整的主工作区，已经有助手类型列、话题列和右侧主内容区三段布局。
- 右侧聊天区域目前还是“按配置状态显示不同提示的骨架面板”，不是完整聊天 UI。
- `settings` 工作区已经有较完整的设置页外观和交互反馈，但大部分字段仍然只是前端本地状态。
- `capabilities`、`files`、`developer` 三个工作区现在主要用于表达未来信息架构，仍属于占位工作区。

## 再展开说明

### 整体界面结构：先有工作台，再逐步填功能

当前主界面最容易理解成一个桌面工作台：左边是窄导航栏，右边根据当前选择切换不同工作区。

现在可以切换的工作区有五个：

- `assistant`
- `capabilities`
- `files`
- `developer`
- `settings`

这说明前端已经先把整体信息架构搭起来了。换句话说，现在最成熟的是“页面分区和使用路径”，不是“每个分区都已经接通真实业务”。

### 左侧窄栏：负责切工作区，不负责业务本身

左侧窄栏当前承担三件事：

1. 在主要工作区之间切换
2. 把 `settings` 单独放在次级位置
3. 维持桌面工作台的导航方式

这里的重点是：**导航结构已经成立**。但导航结构成立，不等于被点进去的每个工作区都已经有真实数据能力。

## `assistant` 工作区：现在最接近主路径的区域

### 这一块现在长什么样

`assistant` 当前采用三段布局：

1. 左列：助手类型
2. 中列：话题或会话列表
3. 右列：主内容区

从界面完成度来看，这一块是目前最值得继续往下接真实功能的地方，因为它已经有比较稳定的交互骨架。

### 左列：助手类型是前端固定列表

当前助手类型固定为三类：

- General
- Blackboard
- TIS

这里的“助手类型”可以理解成“按能力域分的入口分组”。它们已经能点，也会切换中间的话题列表，但当前仍然是前端静态定义，不是从后端动态拉回来的助手注册表。

所以这一列的准确说法是：

- **当前事实**：前端导航结构已实现，切换交互已实现
- **不是当前事实**：已经存在真实的动态助手发现机制

### 中列：话题列表能切，但内容是静态示例

中间这列按助手类型显示不同的话题。每个话题有标题、摘要、更新时间和状态标签。

这让界面已经具备“会话切换”的感觉，但目前这些数据仍然是本地静态常量。也就是说：

- 当前可以点不同话题
- 当前可以看到不同的标题和摘要
- 但这些并不是后端同步回来的真实历史会话

所以这里应理解成“会话列表骨架已经有了”，而不是“会话系统已经接通了”。

### 右列：聊天区域现在还是状态面板

右侧主内容区挂的是 [CopilotChatPanel()](../../frontend-copilot/src/features/copilot/CopilotChatPanel.tsx#L18)。

这里最容易被误解，所以先直接说结论：**当前不是完整聊天窗口，而是一个根据配置状态切换内容的说明面板。**

现在它会根据运行态显示不同内容：

- `loading`：正在读取配置
- `empty`：还没配好最小连接信息
- `incomplete`：只填了一部分
- `ready`：最小配置齐了，骨架可继续往下接
- `error`：读取配置失败

如果你想看每个状态的触发条件和排查建议，去看 [reference-runtime-states.md](./reference-runtime-states.md)。

### 右列里当前已经有的东西

当前已经实现的，是这些：

- 启动阶段由根层统一读取本地 Copilot settings/runtime 摘要
- 根据配置状态切换显示不同面板
- 在 `ready` 时展示根层注入的 `runtimeUrl`、`agentName` 和存储状态
- 在 `empty` 和 `incomplete` 时明确提示“还没有连上后端智能体”
- 在 `error` 时明确提示“这是读取链路异常，不只是没填配置”

### 右列里当前还没有的东西

当前还没有这些真实聊天能力：

- 消息列表
- 输入框发送对话
- 流式返回展示
- 会话持久化
- 与运行时互动后的聊天反馈 UI

因此，当前最准确的表述不是“聊天功能已完成”，而是“聊天面板骨架已就绪”。

## `settings` 工作区：外观和交互已经很多，但多数还是前端本地状态

### 这一块为什么容易误解

`settings` 是当前第二个比较完整的工作区。它已经有独立布局，不再复用 `assistant` 的助手列和话题列语义。

它包含：

- 左侧设置导航列
- 右侧设置主内容区

设置分区目前包括：

- 模型服务
- 默认模型
- 常规设置
- 显示设置
- 数据设置
- MCP 服务器
- 网络搜索
- 全局记忆
- API 服务器
- 文档处理

问题在于：这一页“看起来很像已经能用了”，但里面大多数内容现在还只是前端演示交互。

### 当前已经实现的，是界面交互和信息架构

设置页目前已经有很多可操作的界面反馈，比如：

- 服务商列表切换
- 新增服务商
- 输入框编辑
- 下拉选择
- 开关切换
- 不同设置分区之间切换

这意味着前端已经把“设置中心该长什么样”这件事做得比较具体了，后面继续接真实配置时，不需要再从零设计界面。

### 当前大部分字段只是前端本地状态

设置页里的很多内容当前都由 React 本地 state 驱动。白话来说，就是：

- 改了会立刻看到界面变化
- 但这些值大多只存在当前页面运行时里
- 它们现在大多不会成为真正生效的持久化配置
- 也不能当成已经和后端约好的正式接口字段

这类内容包括但不限于：

- 模型服务商及其详情
- 默认模型路由
- 常规设置中的语言、代理、通知
- 显示设置中的主题、字号、动画
- 数据、MCP、搜索、记忆、API、文档处理中的大多数字段

### 当前真正接到底层链路的范围很小

基于现有代码，真正已经接到 Electron 本地配置链路、并参与 Copilot 启动判断的字段，只有两个：

- `runtimeUrl`
- `agentName`

但要特别注意：**当前设置页界面本身并没有提供这两个字段的正式编辑入口。**

也就是说，现状不是“设置页已经完整接通，只是文档没写”，而是：

- 底层读取 / 保存能力有了
- 根层会统一读取这两个字段并将状态注入聊天面板
- 应用启动时会用这两个字段判断是否进入 `ready`
- 但设置页里目前展示的大量字段，绝大多数与这条真实生效链路并不相连

如果你想直接查字段事实，请看 [reference-current-fields.md](./reference-current-fields.md)。

## `capabilities`、`files`、`developer`：都有独立入口，但目前还是占位工作区

### `capabilities`

这一页现在更像“能力中心的布局草图”。它展示了能力分组、亮点和条目，但这些内容当前并不是实时能力检测结果。

准确说法是：

- **当前事实**：有独立工作区、有清晰的能力分类结构
- **前端占位**：还没有真实工具状态、能力注册结果或调用反馈

### `files`

这一页现在更像“文件工作区以后可能怎么组织”的展示。页面上能看到课程资料、个人笔记、对话附件等方向，但还没有真实文件索引、上传结果或知识库数据接入。

准确说法是：

- **当前事实**：文件工作区结构和方向已经表达出来
- **前端占位**：还没有真实文件数据和导入链路

### `developer`

这一页现在更像“未来开发工作台”的概念入口。它表达了任务、构建验证、变更历史等方向，但还没有真实日志、任务状态或调试面板。

准确说法是：

- **当前事实**：开发工作区入口和结构已出现
- **前端占位**：真实开发数据还没接进来

## 可以把当前界面分成三类内容

| 类别 | 当前状态 | 应怎样理解 |
| --- | --- | --- |
| 工作区切换、助手切换、会话切换、设置页切换 | 当前事实 | 这些交互和页面结构已经存在 |
| 聊天区域的状态提示、设置页多数字段、三个扩展工作区内容 | 前端占位 / 前端本地交互 | 有界面、有反馈，但大多还没持久化，也没接后端 |
| 未来更完整的聊天、真实能力中心、真实文件中心、真实开发工作台 | 未来方向 | 可以讨论，但不能写成已实现 |

## 当前边界 / 不要误解的地方

- 当前界面已经“像一个产品”，不等于当前能力已经“像一个完整产品”。
- `assistant` 的右侧主区域现在还不是完整聊天窗口，只是按配置状态切换的骨架面板。
- 设置页里很多字段可以改，不等于这些字段已经保存、已经生效、或者已经有后端接口支持。
- `capabilities`、`files`、`developer` 现在仍然主要是结构占位，不应写成真实业务页面。
- 当前真正能当作后端连接事实依赖的内容非常有限，详细边界请以 [backend-connection-contract.md](./backend-connection-contract.md) 为准。
