# 前端当前 UI 状态

说明用户在桌面端看到的工作区、页面及其角色。字段位置见[当前生效字段参考](./reference-current-fields.md)，页面成熟度见[页面能力参考](./reference-page-capabilities.md)。

## 工作台总体结构

前端是运行在 Electron 宿主中的 React renderer。进入主工作台后，整体是"左侧入口栏 + 右侧主内容区"的布局。

左侧入口栏包含六个工作区：

- 助手。
- 能力。
- 文件。
- 南科（SUSTech）。
- 日历（开发）。
- 设置。

进入主路径的工作区主要是"助手"和"设置"。南科工作区已接入 Blackboard 数据浏览器和同步面板，文件和能力工作区也已在不同程度上接入了真实数据面。

## 启动阶段用户会看到什么

应用启动后，前端进入根装配阶段。界面根据公开配置中心和 hosted runtime 快照判断状态，决定是进入工作台还是停留在说明性状态页。

用户能看到的典型状态包括：

- 正在准备桌面界面。
- 正在连接本地服务。
- 宿主启动失败或读取链路失败后的可解释错误卡片。

这部分界面的重点是把"前端还在装配""宿主正在启动""没有可用运行时地址"这些情况区分开，而不是把所有失败写成同一种连接错误。

## 助手工作区

助手工作区是最清晰的一条真实主路径。页面主体采用三栏结构：

1. 左侧是智能体目录。
2. 中间是会话列表。
3. 右侧是聊天主内容区。

### 左侧智能体目录

左侧目录展示后端返回的智能体列表。前端负责目录展示、选中状态和基础交互；目录内容以后端返回为准。

### 中间会话列表

中间列支持这些动作：

- 为智能体创建会话。
- 在窗口内切换会话。
- 拖动排序会话。
- 打开会话右键菜单。

"先选智能体，再建会话"是聊天主路径的一部分。会话列不是占位壳，但它首先服务当前窗口里的会话操作。

### 右侧聊天主内容区

右侧区域根据连接状态、目录状态和会话状态切换不同界面。进入可聊天分支后，具备完整的消息发送壳：

- 可滚动的消息区。
- 空会话提示。
- 底部输入区。
- 模型选择器。
- 工具选择器。
- 发送按钮。

### 模型选择器展示真实模型路由目录

模型选择器展示的模型目录来自 settings workspace 里的 provider profiles，而不是静态内置默认模型。如果 settings workspace 里还没有任何 provider 或模型，这里显示"尚未配置模型"的空状态，不自动补出示例模型。

用户选中的不是一个单独的模型字符串，而是一条完整模型路由。页面围绕下面这组信息工作：

- `providerProfileId`
- `provider`
- `endpointType`
- `baseUrl`
- `modelId`

关键事实：产品里不存在"单一 active provider 决定全部模型集合"的主线语义。聊天区看到的是跨 provider 的模型目录。

### 工具选择器接入工具生命周期主线

工具选择器展示会话能力面返回的工具目录、推荐工具和默认信息。发送时，前端把用户选择的工具 ID 显式带进请求。

主线已落地三层边界：目录由 `capabilities/get` 下发，请求通过 `enabledTools` 传启用工具 ID，工具调用过程通过同一条 run 事件流里的 `tool_event` 回传。工具步骤直接出现在聊天消息流里，不再停留在诊断信息或终态汇总字段。

### 聊天区按流式 run 过程更新

聊天区不再等待整包响应后一次性落屏。页面按 run 生命周期工作：

- 收到 `run_started` 时创建 assistant 占位项。
- 收到 `tool_event` 时插入或更新工具步骤，按 `started`、`completed`、`failed` 渲染状态。
- 收到 `text_delta` 时持续增量更新 assistant 文本。
- 收到 `run_completed` 时定稿最终 assistant 文本，记录最终解析出的模型路由。
- 收到 `run_failed` 或 `run_cancelled` 时进入对应终态，未完成工具步骤收口为失败或取消显示。
- 收到 `run_diagnostic` 时保留非敏感诊断信息。

用户看到的聊天响应是实时流式推进的过程，不再是旧"一次请求得到完整 assistant 文本"界面。

### 取消与失败在页面上的表现

页面保留失败项或取消态提示，用于当前窗口里的可见反馈。正式会话归档只在成功完成时发生。页面上的失败项和取消态草稿不代表后端已把 assistant 消息写入正式历史。

## 设置工作区

设置工作区是固定导航下的完整工作区。左侧导航包含 10 个分区：

1. `SUSTech 信息`
2. `模型服务`
3. `默认模型`
4. `常规设置`
5. `显示设置`
6. `API 服务器`
7. `搜索设置`
8. `MCP 设置`
9. `文档处理`
10. `外部源`

这组导航说明两件事：设置页覆盖范围已经明显扩大，页面结构已经稳定到可以承载长期设置。

### 最成熟的设置分区

`模型服务` 是最完整的设置页之一。用户可以看到服务商列表、搜索、添加、复制、删除、拖动排序、详情编辑、API 密钥输入，以及模型列表和模型编辑弹层。

`默认模型` 聚焦两条默认路由：主助手模型和快速执行模型。聊天发送区会从 settings workspace 里读取默认模型路由。

`显示设置` 提供主题入口，控制桌面端外观主题。

`API 服务器` 同时承载两类内容：公开配置中心里的开发态运行时覆盖地址，以及 settings workspace 里的地址草稿、重连策略和轮询字段，是一个混合页。

### 已可见但有边界的分区

`SUSTech 信息` 是正式页面，可以看到校园账号信息、CAS 密码输入、Blackboard 相关偏好，以及 `TIS 信息` 区域。页面已存在，但不是每一块都已接成成熟业务链路。

`外部源` 有真实入口。用户可以看到 WakeUP 分享链接输入框、解析按钮和解析结果弹窗。流程主要停留在前端侧的半接通状态。

`常规设置`、`搜索设置`、`MCP 设置` 和 `文档处理` 已有稳定页面与表单结构，作为长期设置工作区使用，不全是前端主路径里的强生效项。

## 其余工作区状态

`能力`、`文件`、`南科` 和 `日历` 四个工作区具备稳定的 Hub 页面布局，各自已在不同程度上接入了真实数据面。

`南科` 工作区走得更远：接入了 Blackboard 数据浏览器（课程、公告、作业、成绩、资源）、同步状态面板和同步控制入口。

`能力` 工作区具备 MCP 服务器管理（添加、编辑、删除、测试连接）、技能面板和工具权限管理。

`文件` 工作区具备完整文件树操作（CRUD、拖拽、上下文菜单、键盘导航）。

`日历` 工作区具备看板视图、Gantt 图和 Timeline 视图。

按各自的实际成熟度分别描述各个工作区，而不是整体归为"结构占位"。

## UI 常见注意点

- 设置页有很多字段能够保存，不代表都已经接到真实运行时。
- 聊天区有模型选择器，不代表前端内置了一套默认 provider 或默认模型。首次状态可以是空白模型目录。
- 聊天区是流式渲染主线，不再是消费非流式完整响应。
- `外部源` 页面已经存在，不代表 WakeUP 已经打通稳定后端流程。
- `能力`、`文件`、`南科` 和 `日历` 有页面且具备真实功能，但数据回流成熟度仍在持续演进。
- 仓库里保留的 `copilot` 命名主要是历史内部命名空间，不意味 UI 主线仍建立在第三方 CopilotKit runtime 上。
