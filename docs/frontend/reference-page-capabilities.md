# 页面能力参考

## 文档用途

这份附录用于快速回答：**五个工作区当前各自的数据从哪里来、交互做到什么程度、有没有持久化、有没有接后端，以及下一步更可能往哪里补。**

它适合在以下场景使用：

- 新同学快速判断各页面成熟度
- 评审时核对“这页到底是事实、占位，还是未来方向”
- 规划开发顺序时确认哪些页面更接近主路径

## 使用边界

- 本文只描述当前页面能力，不补写不存在的数据源或接口。
- 本文中的“是否接后端”是按当前代码事实判断，不按页面外观判断。
- 本文中的“下一步方向”只表示合理的后续补齐方向，不表示已经实现。
- 如果你想查字段或运行态，请分别看 [reference-current-fields.md](./reference-current-fields.md) 和 [reference-runtime-states.md](./reference-runtime-states.md)。

## 必要说明

### 1. 为什么 `assistant` 写成“部分持久化、部分接后端”

因为当前只有 `assistant` 右侧聊天骨架这条主路径，会真正读取本地 Copilot 配置，并在 `ready` 时把 `runtimeUrl` 和 `agentName` 交给 Copilot 外层能力。

但这并不代表整个 `assistant` 工作区都已经接通：

- 左侧助手类型是静态列表
- 中间话题列表是静态列表
- 右侧仍然是状态骨架，不是完整聊天产品

所以它既不能写成“完全占位”，也不能写成“完整联调页面”。

### 2. 为什么 `settings` 看起来最完整，却仍写成“未接后端”

因为“页面很完整”和“配置已经生效”是两回事。

当前 `settings` 页面的确有很多可交互元素，但这些元素大多数只是前端本地 state，不会直接进入当前已生效的启动链路，也没有形成明确的后端契约。

所以这页当前最准确的定位是：

- **当前事实**：设置中心的界面结构和交互外观已经比较完整
- **前端占位**：多数设置项还没有真正持久化，也没有真正接后端

### 3. 为什么后三个工作区都写成“低交互、未持久化、未接后端”

因为当前 `capabilities`、`files`、`developer` 的共同特征很明显：

- 有独立入口
- 有完整页面结构
- 有标题、说明、亮点和条目
- 但内容主要来自前端静态常量
- 没有真实数据回流
- 没有真正的业务操作闭环

所以它们当前更像“未来页面的结构草图”。

## 当前已实现 / 未实现说明

### 当前已实现

- 五个工作区都已有独立入口
- `assistant` 和 `settings` 的页面完成度明显高于其余三个工作区
- `assistant` 已有最小连接状态驱动的聊天骨架
- `settings` 已有较完整的设置页组织方式和交互外观
- `capabilities`、`files`、`developer` 已有清晰的页面主题和结构雏形

### 当前未实现

- `assistant` 的真实消息流、真实会话历史和完整聊天 UI
- `settings` 中大多数配置项的统一持久化和正式生效
- `capabilities` 的真实工具状态和能力调用结果
- `files` 的真实文件数据、导入链路和知识索引
- `developer` 的真实日志、任务、验证结果和调试面板
