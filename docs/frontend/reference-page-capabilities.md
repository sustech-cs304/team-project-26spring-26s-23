# 页面能力参考

这份附录适合在两种场景下使用：

- 新同学快速判断“这页现在到底做到哪一步了”。
- 评审文档时核对“这页是正式能力、局部接通，还是仍然是占位结构”。

## 先给结论

当前五个工作区的成熟度并不一样：

- **`assistant`**：已经进入真正主路径，重点在“后端目录 → 会话 → 消息发送”。
- **`settings`**：页面最完整，但并不代表所有表单都已经进入正式配置闭环。
- **`capabilities`、`files`、`developer`**：仍然主要是结构化占位页。

所以判断页面成熟度时，不能只看界面完整度，还要看三件事：

1. 数据从哪里来。
2. 改动会不会真正进入系统主链路。
3. 页面里的操作会不会影响真实运行行为。

## 总表

| 工作区 | 当前数据来源 | 当前交互程度 | 是否正式持久化 | 是否接后端 / runtime | 当前最重要的事实 | 下一步方向 |
| --- | --- | --- | --- | --- | --- | --- |
| `assistant` | 智能体目录来自后端 `agents/list`；会话列表来自当前窗口内存状态；聊天区消费根装配层 bootstrap 状态与后端会话能力面 | 高：可拉目录、选智能体、创建会话、切换会话、发送消息、切换模型和工具 | 部分：聊天依赖的 `runtimeUrl` 等字段来自配置中心；会话列表本身仍不持久化 | 是：当前已接 runtime，并进入 session-first 主路径 | 当前主路径已经不是旧全局 agent 方案，而是“目录 → 会话 → 消息” | 继续补历史回放、会话恢复与更完整消息体验 |
| `settings` | 少量正式字段来自统一配置中心；大量表单仍来自 React 本地 state | 高：导航、输入、下拉、开关、服务商列表编辑、模型编辑都可操作 | 部分：`theme`、`animationsEnabled`、`agentName`、`runtimeUrl`、`model` 已正式持久化 | 部分：主题和动画立即影响 UI，`runtimeUrl` 影响连接判断，`model` 影响下次 runtime 启动 | 页面很完整，但正式接入范围仍然是少量关键字段 | 继续按 owner 清楚、确实需要长期生效的字段逐步接入 |
| `capabilities` | 前端静态文案和结构化条目 | 低：主要是浏览与切换页面 | 否 | 否 | 当前主要承担未来能力页的信息架构 | 后续可接真实工具目录、权限状态和调用反馈 |
| `files` | 前端静态文案和结构化条目 | 低：主要是浏览与切换页面 | 否 | 否 | 当前主要承担未来文件工作区的信息架构 | 后续可接文件索引、附件管理和资料导入 |
| `developer` | 前端静态文案和结构化条目 | 低：主要是浏览与切换页面 | 否 | 否 | 当前主要承担未来开发工作台结构 | 后续可接任务、构建、日志和验证记录 |

## 逐页怎么理解

### 1. `assistant` 为什么已经算主路径

当前助手工作区最重要的变化，是它已经不再是“左边静态助手列表 + 右边状态卡片”的旧理解方式。

现在它已经分成三块：

1. **左侧智能体目录列**
   - 数据来自后端 `agents/list`
   - 前端只是做图标和文案增强
   - 真源在后端，不在前端静态常量

2. **中间会话列**
   - 用户可以创建会话
   - 会话列表保存在当前窗口内存里
   - 可以切换当前激活会话

3. **右侧聊天区**
   - 只有在已有可用 runtime URL 时才继续装配
   - 会在当前会话壳下显示消息区、模型选择器、工具选择器和输入区
   - 通过 `message/send` 发真实请求

#### 当前已经真正接通了什么

- 后端智能体目录
- 创建会话
- 读取会话能力面
- 发送消息
- 会话级工具选择
- 请求级模型选择

#### 当前还没有做到什么

- 会话列表跨刷新恢复
- 自动回放后端完整历史
- 更完整的消息元数据展示
- 更成熟的多会话管理体验

所以它现在最准确的定位是：

- **已进入正式聊天主路径**
- 但**还不是完整聊天产品终态**

### 2. `settings` 为什么要特别小心阅读

设置页是当前最容易让人误判的地方。

因为它看起来最完整，但“可操作”不等于“已经正式接到系统主链路”。

#### 当前已经正式接入的部分

- **显示设置**
  - `theme`
  - `animationsEnabled`

- **常规设置**
  - `agentName`

- **默认模型**
  - `model`，通过后端模型卡片写入配置中心

- **API 服务器**
  - `runtimeUrl`
  - 根装配层状态摘要
  - 统一 retry 动作

#### 当前仍主要是页面交互的部分

- 模型服务商列表与模型编辑器
- 默认模型页中的主助手模型 / 快速执行模型路由
- 常规设置里的语言、代理、通知、自动备份
- 显示设置里的字号和界面密度
- 数据、MCP、搜索、记忆、文档处理页面的大多数表单
- API 页面里的后端地址、重连策略、健康检查轮询

因此当前更准确的写法应当是：

- 设置页已经开始承载正式配置入口；
- 但远没有整页全部进入统一配置中心。

### 3. `capabilities`、`files`、`developer` 为什么仍算占位页

这三页当前有几个共同特点：

- 有独立入口
- 有页面结构
- 有标题、亮点和条目
- 但内容主要来自前端静态文案
- 没有真实数据回流
- 也没有正式持久化闭环

所以它们当前的价值更像是：

- 给未来真实能力页预留信息架构和布局容器

而不是：

- 已经接到配置中心或 runtime 的正式业务页

## 页面能力现在最容易被误写错的地方

### 不要再把 `assistant` 写成“左侧静态列表主导”

当前左侧智能体目录已经来自后端，而不是前端静态真源。

### 不要再把 `assistant` 的中间列写成“纯占位”

当前中间列已经承担：

- 创建会话
- 展示当前窗口中的会话列表
- 切换激活会话

虽然它还没做到持久化恢复，但已经不是空壳。

### 不要把 `settings` 的完整外观写成“全部已生效”

这仍然是当前最常见误判来源。

### 不要把扩展工作区写成“已接后台能力页”

当前 `capabilities`、`files`、`developer` 还没有进入真实业务闭环。

## 快速结论

### 当前已进入正式主路径的页面

- `assistant`
- `settings`（仅限其中少量正式字段和状态入口）

### 当前主要是结构承载的页面

- `capabilities`
- `files`
- `developer`
