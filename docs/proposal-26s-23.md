# Proposal Team 26s-23

## Part I. Preliminary Requirement Analysis

### 1. 项目概述

本项目拟开发一个面向 SUSTech 学生的 Student Productivity Agent。系统定位不是替代 Blackboard、日历、学校官网或笔记软件等成熟应用，而是作为一个智能收集、处理与导出中枢：从多个外部来源收集与整合信息，使用智能体理解用户目标，调用合适工具完成规划、检索与分析，并以清晰、可解释、可导出的方式返回结果。

与传统待办应用或固定流程脚本不同，本系统遵循 Agentic Loop 模式。用户通过自然语言提出需求后，系统根据上下文自主决定调用哪些工具、以何种顺序处理数据，并在需要时继续观察结果和调整策略。为满足课程对 Student Productivity Agent 赛道的要求，系统将提供图形用户界面，用于对话交互、结果展示、过程轨迹呈现以及高风险操作确认。

本项目重点解决学生在多平台、多资料、多任务并行场景下的信息割裂问题。系统将优先连接 Blackboard、个人日历或 ICS、学校官方文档与网页、本地学习资料等外部信息源，在尽量不改变用户原有软件使用习惯的前提下，帮助用户更高效地收集信息、形成计划、获取答案并导出可执行结果。

### 2. 功能需求

为保证系统具有足够的功能完整性和 feature 正交性，本项目将围绕以下六项高层功能展开设计与实现。

#### 2.1 多源日程规划

系统应支持用户以自然语言描述学习目标与时间安排需求，并由智能体自动整合 Blackboard 截止日期、固定日程和个人待办等多种时间信息，生成尽量无冲突、可执行的学习安排方案。该功能的重点不只是展示已有事件，而是让智能体在多个时间来源之间进行统一理解、冲突检测和任务排布。

#### 2.2 智能体思维轨迹与用户确认

系统应向用户展示智能体在执行任务时的关键过程，包括正在调用的工具、调用目的、执行状态以及失败或等待确认的原因。同时，对于可能影响本地数据、外部服务或用户日程的高风险操作，系统应提供明确的用户确认机制，使用户在关键动作发生前拥有最终决定权。

#### 2.3 图形化智能体交互工作台

系统应提供完整的图形用户界面，支持用户与智能体进行对话、查看执行结果、浏览日志信息并管理基础配置。该工作台不仅是聊天窗口，还应承担结果呈现、过程反馈以及后续功能扩展入口的作用，从而满足课程对 GUI 的要求，并提升整体可用性。

#### 2.4 学习资料辅助分析

系统应支持用户导入课程相关资料，如课堂笔记、讲义或作业文档，并由智能体对这些内容进行分析，生成摘要、复习题目或启发式提示。该功能面向课程学习场景，重点在于帮助学生更高效地理解和复习材料，而不是直接给出机械化答案。

#### 2.5 校园官方信息问答

系统应支持用户就学生手册、学籍规定、场地预约规则等校园官方信息进行提问，并通过检索和组织官方文档内容生成回答。回答应尽量附带来源依据，以提高结果的可信度，并帮助用户快速定位相关制度条款。

#### 2.6 可视化日程审阅与协同编辑

系统应支持将智能体生成的日程方案以可视化方式展示给用户，并允许用户在界面中对日程进行人工调整。调整后的结果应能够继续与智能体联动，用于进一步冲突检测、重新规划或版本比较，从而形成 AI 生成与用户修正相结合的协同式排期流程。

### 3. 非功能需求

除功能完整性外，本系统还应满足以下非功能需求，以保证其在真实学生场景中的可用性、可信性和可持续性。

#### 3.1 可用性

系统应提供清晰、直观的图形用户界面，使用户能够通过自然语言方便地提出任务、查看执行结果，并理解智能体当前的工作状态。对于日程安排、资料分析和校园问答等核心功能，界面应尽量降低学习成本，并提供一致的交互逻辑。

#### 3.2 可解释性与可审计性

系统应向用户展示智能体的关键执行过程，包括工具调用、执行状态、失败原因和等待确认的信息，使用户能够理解系统为何产生某项结果。该要求与课程对思维轨迹展示的要求一致，也有助于提升用户对系统的信任度。

#### 3.3 安全性

对于可能影响本地文件、外部服务或用户日程数据的高风险操作，系统应提供明确的确认机制，避免智能体在未经授权的情况下直接执行修改类行为。系统还应限制高风险操作的触发范围，降低误操作带来的损失。

#### 3.4 隐私与数据保护

系统在处理 Blackboard 信息、个人日程、课程资料和校园文档时，应尽量减少敏感信息的暴露范围。用户凭证、访问令牌及本地缓存数据应通过安全方式存储，不应在界面或日志中明文泄露。对于不再需要的本地数据，应支持清理或替换。

#### 3.5 性能

系统应在合理时间内完成核心任务。对于常见请求，如读取近期 Blackboard 截止日期、生成简要学习计划或回答校园制度查询，系统应保持可接受的响应速度，避免因过长等待影响使用体验。对于文档分析和检索任务，系统应通过缓存、索引或分阶段处理来控制延迟。

#### 3.6 可靠性

系统在外部接口失败、网络异常、数据缺失或工具执行错误时，应能够返回清晰的错误提示，而不是直接崩溃或输出无意义结果。对于 Blackboard、日历订阅或文档检索等依赖外部数据源的功能，系统应具备基本的错误恢复与降级能力。

#### 3.7 可维护性与可扩展性

系统应采用模块化设计，使外部平台接入、日程规划、文档处理、校园问答和图形界面等模块可以相对独立地开发和演进。系统还应保留后续接入更多成熟应用接口和更多导出方式的扩展空间，以支持未来需求变化。

#### 3.8 兼容性

系统应至少支持课程项目当前使用的桌面端开发环境与主流开发机操作系统，并保证在所选前后端框架下能够稳定运行。不同模块之间的数据接口应尽量保持清晰，以便后续接入更多校园数据源、资料格式或外部服务。

### 4. 技术需求

本项目的技术路线强调连接外部成熟应用与信息来源，并通过统一的智能体架构完成信息处理与结果导出，而不是完全自建所有数据与服务。

#### 4.1 运行环境

系统应运行在具备图形界面的个人计算机环境中，初期面向课程项目常用的桌面开发环境与学生个人电脑。系统需要具备稳定的网络访问能力，以连接 Blackboard、学校官网、日历订阅链接及其他外部信息源。

#### 4.2 系统架构

系统应采用 Agentic Loop 架构，支持自然语言输入、任务理解、工具选择、工具调用、结果观察与用户确认等完整流程。前端图形界面负责展示对话、结果、轨迹与确认交互；后端智能体服务负责推理、调度工具、整合多源数据并生成可解释输出。

#### 4.3 前端技术栈

前端应采用基于 TypeScript、React 和 Electron 的桌面端图形界面方案，支持聊天交互、结果展示、日志侧栏、确认弹窗以及后续资料上传、日历编辑等扩展能力。界面层应便于承载多种输入输出形式，而不仅仅是单一聊天窗口。

#### 4.4 后端技术栈

后端应采用 Python 3.12 作为主要开发语言，结合智能体编排框架、网页请求与抓取工具、文本处理工具和数据持久化组件，实现 Blackboard、日历订阅、校园文档和本地文件等数据源的统一处理。后端应具备模块化工具封装能力，以便后续持续接入新的外部接口，而不需要重写整体系统。

#### 4.5 数据存储与中间表示

系统应使用本地数据库或结构化文件保存必要的缓存数据、索引数据、会话状态和工具输出结果。为支持多源整合，系统需要定义统一的数据中间表示，例如事件数据、文档片段、问答结果和导出结果，以降低不同外部来源之间的耦合。

#### 4.6 外部接口集成能力

系统的一个关键技术要求是能够连接外部成熟应用或信息服务，而不是完全替代它们。首要集成对象包括 Blackboard、ICS 或个人日历、学校官方网页或文档，以及本地课程资料。系统架构应保留后续接入更多搜索接口、知识来源或第三方应用的空间。

#### 4.7 导入与导出能力

系统不仅需要接收外部信息，还需要将处理结果以用户可用的形式导出。导入方面，系统应支持从外部平台、订阅链接或本地文件接收结构化与非结构化数据。导出方面，系统应支持将排期结果、资料分析结果、问答结果等整理为可展示、可保存、可继续传递给其他模块或外部服务的格式。

#### 4.8 可测试与可扩展实现

系统应便于进行模块测试、接口测试和端到端联调，以保证在多个外部接口存在波动的情况下仍可持续迭代。系统中的工具、数据模型和界面模块应保持清晰边界，支持后续增加新能力而不破坏现有功能。

### 5. 数据需求

本系统的数据需求以多源收集、统一表示、可信处理和结果导出为核心，强调在真实学生场景下对外部信息源的有效整合，而不是依赖单一封闭数据体系。

#### 5.1 Blackboard 学习数据

系统需要获取与学生学习安排直接相关的 Blackboard 数据，包括课程作业、截止时间、公告、课程资源链接以及可用的课程日历信息。这些数据主要通过 Blackboard 接口或网页抓取方式获取，并在本地进行结构化整理，用于后续的日程规划与任务提醒。

#### 5.2 个人时间安排数据

系统需要获取用户已有的固定时间安排，例如课程时间、组会、考试、个人事务和其他不可移动事件。这类数据可以来自 ICS 订阅、本地日历导出文件或用户手动输入。系统需要将这些数据统一转换为标准事件格式，以便与 Blackboard 截止时间和学习计划一起进行冲突检测与排期。

#### 5.3 校园官方文档与网页数据

系统需要接入学校官方信息来源，如学生手册、学籍管理规定、教学安排通知、场地预约说明和其他正式发布的网页或文档。这些数据用于支持校园信息问答功能。为保证可信度，系统需要尽量保留来源信息，如标题、章节、条款编号和原始链接。

#### 5.4 学习资料与本地文件数据

系统需要支持用户导入课程相关资料，例如 Markdown 笔记、PDF 讲义、作业文档以及后续可能扩展的课件文件。这些数据将用于生成摘要、测验、提示和复习材料。对于非结构化文档，系统需要先完成文本抽取、清洗和切分，再进入后续分析流程。

#### 5.5 用户输入与交互上下文数据

系统需要记录用户的自然语言请求、任务目标、当前对话上下文以及用户对高风险操作的确认结果。这些数据用于支持智能体连续理解当前任务、跟踪执行状态，并在必要时向用户返回更符合上下文的结果。对于此类数据，系统应尽量限制保留范围，避免无必要地长期保存敏感内容。

#### 5.6 过程日志与轨迹数据

为了满足可解释性与可审计性要求，系统需要保存适量的过程数据，例如工具调用记录、执行状态、错误信息和用户确认结果。这些数据既服务于前端轨迹展示，也服务于调试与系统维护。

#### 5.7 中间处理数据与统一表示

由于系统需要对接多个外部来源，因此需要定义统一的中间数据表示，例如统一的事件对象、文档片段对象、问答结果对象和导出结果对象。这样可以降低 Blackboard、日历、官方文档和本地文件之间的格式差异带来的耦合，并提高系统扩展性。

#### 5.8 结果输出与导出数据

系统不仅需要输入数据，也需要生成可供用户查看、保存或继续使用的结果数据。这些结果包括结构化日程、任务清单、学习摘要、测验题目、提示信息、问答结果以及可导出的摘要报告或结构化结果文件。导出形式可以是界面展示、文本报告或可继续传递给其他模块与外部服务的标准格式。

#### 5.9 数据更新与获取策略

系统需要考虑不同数据源的更新频率与获取方式。对于 Blackboard 截止时间与日历事件，应支持定期刷新或按需重新获取；对于校园官方文档，应支持版本替换与索引更新；对于用户本地资料，则应支持手动重新导入与覆盖旧结果，从而避免系统长期依赖过期数据。

#### 5.10 数据安全与最小化原则

系统应仅收集完成当前任务所需的必要数据，并优先采用本地缓存和本地存储方式处理敏感内容。对于用户凭证、访问令牌、个人日程和学习资料，应避免在日志或界面中直接暴露敏感细节，并尽量减少不必要的长期保存。
